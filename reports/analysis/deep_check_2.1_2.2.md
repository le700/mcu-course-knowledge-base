**需求原文：** "某方向绿灯或黄灯亮时，另一方向保持红灯。"

**验证方法：** 逐一检查每个相位下两个方向的灯光输出状态，以及所有特殊模式下的互锁情况。

| 软件流程图 图五 | 状态机完整流转图 | 07_软件流程图_mermaid.md:115-134 |

#### 逐相位互锁验证：

| 相位 | NS方向 | EW方向 | 互锁满足？ |
|------|--------|--------|-----------|
| NS_GREEN | 绿灯亮 | 红灯亮 | 是 |
| NS_YELLOW | 黄灯亮 | 红灯亮 | 是 |
| EW_GREEN | 红灯亮 | 绿灯亮 | 是 |
| EW_YELLOW | 红灯亮 | 黄灯亮 | 是 |

代码审查报告确认 ApplyLights 函数（行633-684）负责灯光输出，每个相位下只有一方输出绿灯/黄灯，另一方固定输出红灯。状态机只有 4 个合法状态，不存在两个方向同时绿灯/黄灯的状态。

#### 紧急模式互锁：

需求 2.8 明确："优先方向绿灯亮，另一方向红灯亮"。代码审查报告确认紧急模式优先级最高，由 SelectEmergencyDirection 函数选择优先方向，ApplyLights 输出对应的灯光。**紧急模式下不存在双绿灯。**

#### 夜间模式互锁：

需求 2.9 明确："两方向黄灯同步闪烁（500ms 周期）"。夜间模式下两方向同时黄灯闪烁，**无绿灯亮起**。虽然两个方向同时黄灯，但需求 2.1 的互锁条件针对的是"绿灯或黄灯亮时另一方向保持红灯"，而夜间模式是需求 2.9 的独立功能，两个方向统一处理为黄灯闪烁，且有明确的功能说明。**夜间模式不违反互锁原则，因为没有任何方向亮绿灯。**

#### 过渡状态分析：

状态机仅在 `phaseSec == 0` 时通过 AdvancePhase 推进。相位过渡路径为：
- NS_GREEN → NS_YELLOW（绿灯→黄灯，中间有黄灯缓冲）
- NS_YELLOW → EW_GREEN（黄灯→对方绿灯，中间有黄灯缓冲）
- EW_GREEN → EW_YELLOW（绿灯→黄灯，中间有黄灯缓冲）
- EW_YELLOW → NS_GREEN（黄灯→对方绿灯，中间有黄灯缓冲）

**不存在从绿灯直接切换到对方绿灯的路径**。每次切换都经过黄灯作为缓冲，且 ApplyLights 在每次 SystemTick2ms 调用时都会执行（即每 2ms 刷新一次灯光输出），因此任何过渡状态下都不会出现短暂的"双绿灯"。

#### 代码路径全面检查：

根据代码审查报告和流程图，所有可能的执行路径：
1. 正常模式主循环 → SystemTick2ms → phaseSec递减 → AdvancePhase → ApplyLights → 互锁保证
2. 紧急模式进入 → HandleKeys → SelectEmergencyDirection → ApplyLights → 单方向绿灯
3. 夜间模式进入 → HandleKeys → g_night=1 → SystemTick2ms → 黄灯闪烁 → ApplyLights → 两方向黄灯
4. 紧急/夜间退出 → StartPhase(PHASE_NS_GREEN) → ApplyLights → NS绿灯/EW红灯

**无任何代码路径导致两个方向同时绿灯。**

**✅ 通过**

### 验证点 4：段码表

**需求：** 共阴数码管，P0.0=A, P0.1=B, ..., P0.7=DP

**证据来源：**

| 文档 | 内容 | 路径 |

**分析：**

**(a) 共阴数码管原理：** 共阴数码管的公共阴极接地，段选引脚（A-G, DP）接高电平时对应的 LED 段导通发光。即：段码位=1 → 该段亮。

**(b) 段码引脚映射：**

| P0 位 | 段 | 共阴段码位含义 |
|--------|-----|---------------|
| P0.0 | A | bit0=1 → A段亮 |
| P0.1 | B | bit1=1 → B段亮 |
| P0.2 | C | bit2=1 → C段亮 |
| P0.3 | D | bit3=1 → D段亮 |
| P0.4 | E | bit4=1 → E段亮 |
| P0.5 | F | bit5=1 → F段亮 |
| P0.6 | G | bit6=1 → G段亮 |
| P0.7 | DP | bit7=1 → DP段亮 |

**(c) 段码表正确性验证：**

以数字 0 为例：共阴数码管显示 0 需要 A、B、C、D、E、F 段亮，即 bit0-bit5 = 1，bit6(G)=0，bit7(DP)=0 → 段码 = 0x3F。

以数字 1 为例：需要 B、C 段亮 → bit1-bit2 = 1，其余 = 0 → 段码 = 0x06。

这些与标准共阴段码表一致。

**(d) Bug #6 修复确认：** Bug #6 记录了初始选错 7SEG-MPX2-CA（共阳）的问题，已改为 7SEG-MPX2-CC（共阴）。代码审查报告中"数码管扫描无鬼影"和"0 Error, 0 Warning"的编译结果也间接确认段码表正确。

### 验证点 6：倒计时数值更新频率

**需求：** 倒计时显示"剩余等待/通行时间"

**证据来源：**

| 文档 | 内容 | 路径 |
|------|------|------|
| 软件流程图 图二 | `g_secondTicks++` → `secondTicks >= 500?` → `phaseSec--` | 07_软件流程图_mermaid.md:40-43 |
| 需求规格说明书 三 | "Timer0 中断周期 | 2 ms" | 01_项目需求规格说明书.md:71 |
| 代码审查报告 一 | SystemTick2ms 函数（行1032-1083） | 03_代码审查报告.md:28 |

**分析：**

**(a) 倒计时递减机制：**

SystemTick2ms 每 2ms 被调用一次（由 Timer0ISR 中的 `g_tick2ms++` 触发主循环中的 `g_tick2ms--` 和 `SystemTick2ms`）。

在 SystemTick2ms 中：
- `g_secondTicks` 每次 +1（每 2ms 一次）
- 当 `g_secondTicks >= 500` 时，即经过 500 × 2ms = 1000ms = 1 秒
- 此时 `phaseSec--`（剩余时间减 1 秒）
- 然后 `g_secondTicks` 重置

**(b) 更新频率：** 倒计时数值每 1 秒更新一次。符合倒计时显示的基本预期。

**(c) 显示刷新：** 虽然倒计时数值每 1 秒才变化一次，但 UpdateDisplay 函数在每次 SystemTick2ms 调用时都会执行（每 2ms），实时将 `g_phaseSec` 的值写入 `g_dispBuf`，再由 Timer0ISR 以 125Hz 的频率刷新数码管。因此显示本身是流畅的，只是数值变化步长为 1 秒。

**(d) 数值来源：** `g_phaseSec` 在进入每个相位时由 StartPhase/AdvancePhase 设置（绿色相位调用 CalcGreenTime，黄色相位固定 3 秒），之后每秒递减。显示的是当前相位的剩余时间。

**✅ 通过**
