## 验收标准1: 两组交通灯按"绿-黄-红互锁"规律工作，不能两个方向同时绿灯

- **要求**: 两组交通灯按"绿-黄-红互锁"规律工作，不能出现两个方向同时绿灯。NS_GREEN 时 EW 必须为红灯，EW_GREEN 时 NS 必须为红灯。
- **代码证据**:
  - **状态机设计**（代码审查报告 2.1 节）：正常模式相位流转为 `NS_GREEN → NS_YELLOW → EW_GREEN → EW_YELLOW → (循环)`，状态机图中任何时刻只有一个相位处于激活状态。
  - **ApplyLights 函数**（代码审查报告，行 633-684）：负责根据当前相位输出灯光信号，被标记为"正常"。在正常模式下，该函数仅在单一相位（NS_GREEN 或 EW_GREEN）下点亮对应绿灯。紧急模式下由 SelectEmergencyDirection 确定唯一优先方向。
  - **流程图（图五）**：状态机明确显示四个相位互斥，无任何路径从 NS_GREEN 直接跳转到 EW_GREEN（必须经过 NS_YELLOW），也无并行状态。
  - **代码审查结论**："状态机流转正确，无死锁可能"（3.2 节）。
- **硬件证据**:
  - NS_GREEN: P1.2，EW_GREEN: P1.5（硬件设计说明文档 三、引脚分配表）
  - 两组绿灯由独立 GPIO 引脚控制，I/O 口电平由软件统一设置，不存在硬件短路导致同时点亮的风险。
- **风险评估**: **低风险**。状态机设计为严格的顺序流转，正常模式下 NS_GREEN 和 EW_GREEN 不可能同时处于激活状态。紧急模式下 SelectEmergencyDirection 仅选择一个方向放行。潜在风险点在于：如果 ApplyLights 函数实现存在 bug（如位操作错误同时置位 P1.2 和 P1.5），但代码审查报告已确认该模块功能正常，且无功能性问题记录。由于源代码文件未直接提供，无法进行逐行验证，但基于代码审查报告的专业评审结论，该风险极低。
- **判定**: ✅ 通过

## 验收标准2: 两组数码管倒计时稳定显示，无明显闪烁、重影或乱码

- **要求**: 两组数码管倒计时稳定显示，无明显闪烁、重影或乱码。
- **代码证据**:
  - **Bug #1 已修复（重影/鬼影）**：在 Timer0ISR 中断中增加了消隐处理 —— 每次切换位选前先关闭所有位选（`DIG_NS_TENS = 1; DIG_NS_ONES = 1; DIG_EW_TENS = 1; DIG_EW_ONES = 1;`），然后 `P0 = 0x00` 清段码，再送新段码，最后才开启对应位选。代码审查报告确认"数码管扫描无鬼影"（3.2 节）。
  - **Bug #6 已修复（共阴/共阳）**：确认 Proteus 中使用共阴数码管 `7SEG-MPX2-CC`，段码表按共阴方式输出。
  - **Timer0 中断周期**：2ms（需求规格说明书 三、性能指标），流程图（图四）显示 Timer0ISR 每 2ms 刷新一个数码管位。
  - **扫描频率计算**：4 位数码管轮流扫描，每 2ms 刷新 1 位，完整刷新周期 = 4 × 2ms = 8ms，扫描频率 = 1000/8 = **125Hz**。人眼闪烁感知阈值约为 50-60Hz，125Hz 远超此阈值，不会产生可见闪烁。
  - **SetTwoDigits / UpdateDisplay 函数**（代码审查报告，行 686-740）：负责将倒计时数值转换为段码并写入显示缓冲区 g_dispBuf[]，被标记为"正常"。
- **硬件证据**:
  - P0 口输出段码，串联 8×220Ω 限流电阻（硬件设计说明文档 三、P0 口）
  - P2.0-P2.3 为位选信号，低电平有效（P2.0=NS十位, P2.1=NS个位, P2.2=EW十位, P2.3=EW个位）
  - **Bug #2 已修复（P0 口驱动）**：P0 口为开漏输出，外部增加了 8×10kΩ 上拉电阻，确保数码管亮度正常。
- **风险评估**: **低风险**。125Hz 扫描频率对数码管动态显示而言属于标准设计，远高于人眼闪烁阈值。消隐处理已实现，重影问题已修复。共阴/共阳正确匹配。P0 口上拉已添加。唯一需注意：125Hz 是总扫描频率，每个单独数码管位的点亮占空比为 1/4 = 25%，亮度可能略低于静态驱动，但 220Ω 限流电阻配合上拉电阻通常可以提供足够亮度。
- **判定**: ✅ 通过

## 验收标准3: 车流检测按键按下后 OLED 车数和等待车数正确增加，下一轮绿灯时间延长

- **要求**: 车流检测按键（P3.4/P3.5）按下后，OLED 显示的车数和等待车数正确增加，下一轮绿灯时间确实延长。
- **代码证据**:
  - **RegisterVehicle 函数**（代码审查报告 2.2 节，行 911-961）：
    - 累计总车流量：`g_nsFlowTotal` / `g_ewFlowTotal` 递增 ✅
    - 仅当该方向为**红灯**时才计入等待队列：`g_nsVehicleCnt` / `g_ewVehicleCnt` 递增 ✅
    - 等待队列上限为 9（`g_nsVehicleCnt < 9` 限制）✅
  - **CalcGreenTime 函数**（代码审查报告 2.3 节，行 561-582，流程图 图六）：
    - ownCnt 上限 6（防止溢出）✅
    - 基础绿灯时间 = 12 秒 ✅
    - 每辆等待车 +2 秒：`sec = 12 + ownCnt * 2` ✅
    - 本方向车数比对方多 2 辆以上（`ownCnt > otherCnt + 1`）→ 额外 +3 秒 ✅
    - 有行人请求 → +3 秒 ✅
    - 最终限制在 8~25 秒 ✅
  - **下一轮生效机制**：代码审查报告 2.1 节明确指出"NS_GREEN 进入时调用 CalcGreenTime 动态计算绿灯时长"、"EW_GREEN 进入时调用 CalcGreenTime"。因此 CalcGreenTime 在**进入绿灯相位时**被调用，此时读取的是上一轮红灯期间累计的等待车数，从而保证下一轮绿灯时间确实延长。
  - **OLED 显示更新**：OLED_UpdateScreen 函数（行 752-909）使用缓存比较机制（g_oledModeCache 等），仅在值变化时局部刷新，"车数"信息在 OLED 上以 16 像素高大号数字显示。
- **硬件证据**:
  - KEY_NS_CAR: P3.4，KEY_EW_CAR: P3.5（硬件设计说明文档 三、P3 口）
  - 按键消抖：KeyScan 使用 3 次采样确认（DEBOUNCE_CNT = 3），边沿检测（上升沿），长按只触发一次（代码审查报告 2.4 节）。
- **风险评估**: **低风险**。逻辑链路完整：按键 → 消抖 → RegisterVehicle（区分红灯/绿灯）→ 累计等待车数 → CalcGreenTime（下一轮绿灯相位进入时调用）→ 绿灯时间延长。代码审查已确认智能配时算法正确。唯一细微之处：等待车数上限为 9（代码审查报告 3.1 节指出），这意味着最多 9 辆车计入等待队列，对应最多 +18 秒（9×2=18，但 ownCnt 上限为 6，实际最多 +12 秒），这在仿真场景中已足够。
- **判定**: ✅ 通过

## 验收标准4: 行人请求键按下后系统记录请求并缩短对向等待时间

- **要求**: 行人请求键按下后系统记录请求并缩短对向等待时间。对向绿灯时缩短至 5 秒；本向绿灯时保证剩余时间不少于 6 秒。
- **代码证据**:
  - **Bug #8 已修复（行人请求无效）**：原 HandleKeys 中行人处理缺少对当前相位的判断。修复后增加了相位判断逻辑：
    - **对向绿灯时**：缩短对向剩余绿灯时间至 `REQUEST_SHORTEN_SEC`（即 5 秒）✅
    - **本向绿灯时**：延长至 `PED_EXTEND_SEC`（即 6 秒），保证行人过街时间 ✅
  - **HandleKeys 函数**（代码审查报告，行 963-1030）：按键处理优先级为 紧急 > 夜间 > 行人。流程图（图三）显示了紧急和夜间处理，行人处理在夜间检查之后执行（代码审查报告 2.1 节："夜间模式优先级次之，处理后 return，跳过行人处理"）。
  - **行人请求对绿灯时间的影响**：CalcGreenTime 中，若有行人请求（pedReq == 1），绿灯时间额外 +3 秒（代码审查报告 2.3 节）。
- **硬件证据**:
  - KEY_NS_PEDES: P3.3，KEY_EW_PEDES: P3.2（硬件设计说明文档 三、P3 口）
  - 按键按下为低电平（10kΩ 上拉到 VCC，按键接地），KeyScan 上升沿检测确保只触发一次。
- **风险评估**: **低至中等风险**。Bug #8 已明确修复，代码审查报告确认了行人处理逻辑。但存在以下注意点：
  1. 流程图（图三）中未显示行人按键处理分支，仅显示了紧急和夜间处理后的 `return`。但代码审查报告和 Bug 日志均确认行人处理逻辑存在于 HandleKeys 中，流程图可能为简化版本。
  2. 对向绿灯缩短至 5 秒：需要确认 `REQUEST_SHORTEN_SEC` 宏定义为 5，且该值大于黄灯时间（3 秒），逻辑合理。
  3. 本向绿灯保证 >= 6 秒：`PED_EXTEND_SEC` 为 6 秒，确保行人至少有 6 秒过街时间，合理。
  4. 行人请求在 CalcGreenTime 中仅 +3 秒，而需求规格说明书 2.7 节描述的是"缩短对向等待时间"和"保证本向剩余时间"，这是两个不同的机制：前者是即时干预当前相位倒计时，后者是影响下一轮配时。两项机制均已实现。
- **判定**: ✅ 通过

## 验收标准5: 紧急键按下后优先方向绿灯放行、另一方向红灯、数码管显示 EE、蜂鸣器输出

- **要求**: 紧急键按下后优先方向绿灯放行、另一方向红灯、数码管显示 EE、蜂鸣器输出。再次按下退出。
- **代码证据**:
  - **SelectEmergencyDirection 函数**（Bug #7 修复，代码审查报告行 911-961）：
    - 综合考虑三个因素：最近一次红外检测方向、等待车辆数量、当前通行方向 ✅
    - Bug #7 修复前方向选择逻辑不清晰，修复后增加了专有函数 ✅
  - **ApplyLights 函数**（代码审查报告，行 633-684）：紧急模式下，优先方向亮绿灯，另一方向亮红灯 ✅
  - **HandleKeys 紧急处理**（流程图 图三、代码审查报告 2.1 节）：
    - 紧急键按下 → `g_emergency` 翻转 ✅
    - 进入紧急模式（`g_emergency == 1`）→ 调用 `SelectEmergencyDirection` → 设置 `g_oledDue = 1` → 直接 `return`（跳过夜间和行人处理）✅
    - 再次按下紧急键 → `g_emergency` 翻转为 0 → 调用 `StartPhase(PHASE_NS_GREEN)` 恢复正常 ✅
    - 紧急模式优先级最高，确保不被其他模式干扰 ✅
  - **UpdateDisplay 函数**（代码审查报告，行 686-740）：紧急模式下数码管显示 "EE" ✅
  - **蜂鸣器控制**：P2.7 在紧急模式下输出高电平，驱动蜂鸣器鸣响。代码审查报告 ApplyLights 模块标记为"正常"。
- **硬件证据**:
  - KEY_EMERGENCY: P3.0（硬件设计说明文档 三、P3 口）
  - BUZZER: P2.7 → 500Ω → NPN 基极 → 蜂鸣器 → VCC（硬件设计说明文档 五、蜂鸣器驱动电路）
  - P2.7 = 1 时 NPN 导通，蜂鸣器鸣响；P2.7 = 0 时截止，蜂鸣器静音 ✅
  - 蜂鸣器选型为有源蜂鸣器 `BUZZER (DC/Active)`，无需软件产生 PWM 波形 ✅
- **风险评估**: **低风险**。紧急模式设计完整：方向选择有专有函数，灯光输出有 ApplyLights 处理，显示有 UpdateDisplay 处理，蜂鸣器有硬件驱动。Bug #7 已修复方向选择问题。代码审查报告确认紧急模式"优先级最高"且"退出后恢复正常"。需注意：如果 SelectEmergencyDirection 选择的优先方向与当前实际交通状况不符，可能在仿真中看不出问题，但逻辑上已覆盖三个因素，方向选择合理。
- **判定**: ✅ 通过

## 验收标准6: 夜间键按下后两方向黄灯闪烁、数码管显示 --、OLED 显示"夜间"

- **要求**: 夜间键按下后两方向黄灯同步闪烁（500ms 周期）、数码管显示 "--"、OLED 显示"夜间"。再次按下退出。
- **代码证据**:

    - 闪烁频率 = 1000 / (500ms × 2) = 1Hz 完整周期，即 2Hz 翻转频率 ✅
  - **HandleKeys 夜间处理**（流程图 图三）：
    - 夜间键按下 → `g_night` 翻转 ✅
    - 进入夜间模式（`g_night == 1`）→ 直接 `return`，保持夜间状态 ✅
    - 再次按下 → `g_night == 0` → 调用 `StartPhase(PHASE_NS_GREEN)` 恢复正常 ✅
    - 夜间模式优先级仅次于紧急模式（代码审查报告 2.1 节）✅
  - **Bug #4 已修复（退出夜间后状态残留）**：
    - 退出夜间时 `g_flashOn` 置为 1，`g_flashTicks` 清零 ✅
    - 调用 `StartPhase(PHASE_NS_GREEN)` 恢复正常交通灯 ✅
  - **UpdateDisplay 函数**：夜间模式下数码管显示 "--" ✅
  - **OLED 显示**：OLED_DrawLayout / OLED_UpdateScreen 中，夜间模式下显示"夜间"（代码审查报告，行 752-909）。OLED 采用局部刷新 + 缓存比较，避免全屏刷新闪烁（Bug #5 修复）。
- **硬件证据**:
  - KEY_NIGHT: P3.1（硬件设计说明文档 三、P3 口）
  - NS_YELLOW: P1.1，EW_YELLOW: P1.5（硬件设计说明文档 三、P1 口）
  - 两方向黄灯由独立 GPIO 控制，同步闪烁由软件 `g_flashOn` 统一控制 ✅
- **风险评估**: **低风险**。夜间模式设计完整：闪烁频率正确（500ms），Bug #4 已修复退出残留问题，OLED 显示"夜间"已实现，数码管显示 "--" 已实现。需注意：需求规格说明书 2.9 节描述"两方向黄灯同步闪烁"，而代码审查仅提及 `g_flashOn` 统一翻转控制，从流程图看闪烁逻辑为全局控制，可以保证两方向同步。
- **判定**: ✅ 通过
