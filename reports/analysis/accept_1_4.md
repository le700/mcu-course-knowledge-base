### 1.2 动态互锁 — 过渡状态分析

**问题核心：** 状态机切换时，AdvancePhase 先设置 `g_phase = NS_YELLOW`，然后 ApplyLights 再更新 P1 口输出。在这两个步骤之间是否存在短暂的"两个状态叠加"？

**调用链分析（基于流程图 图二）：**

```
SystemTick2ms 调用（每 2ms 一次）:
  → phaseSec-- 
  → phaseSec == 0 ?
    → 是 → AdvancePhase [设置 g_phase = 下一相位, 设置 phaseSec]
    → 否 → (跳过)
  → g_oledDue = 1
  → ApplyLights [根据 g_phase 输出 P1 口]
  → UpdateDisplay
```

**关键发现：** AdvancePhase 和 ApplyLights 都在**同一个 SystemTick2ms 调用**中顺序执行，中间没有中断返回点。具体来说：

1. AdvancePhase 修改 `g_phase`（例如从 NS_GREEN 改为 NS_YELLOW）
2. 随后 ApplyLights 立即读取 `g_phase` 并更新 P1 口
3. 这两个步骤之间执行的是 g_oledDue 赋值（约 1-2 条指令，< 5μs），在此期间 P1 口仍保持上一次 ApplyLights 的输出值

**最坏情况时间窗口：**
- AdvancePhase 修改 g_phase 后，到 ApplyLights 更新 P1 口，中间间隔极短（< 10 条 C 指令，约 10-20μs）
- 在此窗口内，P1 口仍保持旧相位的输出（例如 NS_GREEN 时的 P1.2=1, P1.3=1）
- 这是**旧状态的延续**，而非两个状态的叠加

**结论：** 不存在真正的"两个状态叠加"。在过渡窗口内，P1 口保持的是旧相位的合法输出（如 NS_GREEN），而不是两个相位输出的混合。两个绿灯同时亮的情况不会发生，因为：
- 旧相位（NS_GREEN）输出：P1.2=1(NS绿), P1.3=1(EW红) ← 合法
- 新相位（NS_YELLOW）输出：P1.1=1(NS黄), P1.3=1(EW红) ← 合法
- 过渡期间：先保持旧相位输出，再瞬间切换为新相位输出

| 判定 | 详细说明 |
|:--:|------|
| ✅ | AdvancePhase 和 ApplyLights 在同一个 SystemTick2ms 调用中顺序执行，过渡窗口 < 20μs。过渡期间 P1 口保持旧相位的合法输出，不存在两个绿灯同时亮的情况。 |

### 1.3 中断安全 — g_phase 变量共享分析

**问题：** Timer0ISR 和主循环是否共享 `g_phase` 变量？是否可能导致数据竞争？

**共享变量分析：**

根据流程图和代码审查报告：

| 变量 | 写入者 | 读取者 | 类型推断 |
|------|--------|--------|---------|
| g_phase | SystemTick2ms（主循环上下文） | ApplyLights（主循环上下文） | 枚举（1字节） |
| g_dispBuf[4] | UpdateDisplay（主循环上下文） | Timer0ISR | uint8_t 数组 |
| g_digitIndex | Timer0ISR | Timer0ISR | uint8_t |
| g_tick2ms | Timer0ISR | 主循环 | uint8_t |

**关键发现：g_phase 不在 Timer0ISR 中被读取或写入。**

从流程图 图四（Timer0ISR）可以看到，Timer0ISR 只操作：
- 位选引脚（P2.0-P2.3）
- P0 口（段码）
- g_digitIndex
- g_tick2ms

Timer0ISR 完全不访问 g_phase。因此不存在 g_phase 的并发读写问题。

**g_dispBuf 的并发问题：**
- 主循环中 UpdateDisplay 写入 g_dispBuf
- Timer0ISR 中读取 g_dispBuf
- 这确实存在潜在的并发问题

**分析：**
- Timer0ISR 每 2ms 触发一次，读取 g_dispBuf[g_digitIndex] 并输出到 P0
- 主循环中 SystemTick2ms 调用 UpdateDisplay 写入 g_dispBuf
- 如果 Timer0ISR 在 UpdateDisplay 写入 g_dispBuf 的中途触发，可能读到部分更新的数据

**但是：**
- 8051 是 8 位机，g_dispBuf 每个元素是 1 字节
- 对单个字节的读写是原子的（单条 MOV 指令）
- Timer0ISR 每次只读取 g_dispBuf 的一个元素（g_digitIndex 指向的那个）
- 写入 g_dispBuf[0] 和 g_dispBuf[1] 是两条独立指令，之间可能被打断
- 如果 Timer0ISR 在 g_dispBuf[0] 已更新但 g_dispBuf[1] 尚未更新时触发，且 g_digitIndex 恰为 0，则读取的是新值；若 g_digitIndex 为 1，则读取的是旧值
- 这最多导致某一帧显示中一个位是旧值、一个位是新值，持续 2ms（一帧），人眼无法察觉

**关于 g_phase 类型的原子性：**
- 代码审查报告未直接给出 g_phase 的类型声明
- 从状态机使用场景推断，g_phase 应为枚举类型（如 `typedef enum { PHASE_NS_GREEN, PHASE_NS_YELLOW, PHASE_EW_GREEN, PHASE_EW_YELLOW, PHASE_EMERGENCY, PHASE_NIGHT } Phase;`）
- 枚举在 Keil C51 中默认为 `unsigned char`（1 字节），操作是原子的
- 即使 g_phase 是 struct 的一部分，从代码审查报告的函数职责描述来看，ApplyLights 和 AdvancePhase 各自独立操作 g_phase，不存在通过 struct 指针间接访问的复杂场景

| 判定 | 详细说明 |
|:--:|------|
| ✅ | g_phase 不在 Timer0ISR 中被访问，不存在并发读写竞争。g_dispBuf 存在理论上的读写竞争，但 8051 单字节操作是原子的，且持续仅 2ms，人眼无法察觉。代码审查报告确认"中断保护完善，无竞态条件"（03_代码审查报告.md:111）。 |

### 1.5 夜间模式互锁 — g_flashOn 同步性

**需求：** "两方向黄灯同步闪烁（500ms 周期）"

**代码审查报告证据：**
- 代码审查报告 2.1 节：夜间模式优先级次之
- 流程图（图二）：`g_night == 1` → `g_flashTicks++` → `g_flashTicks >= 250` → `g_flashOn 翻转`

**同步性分析：**

夜间模式下 ApplyLights 的行为推断：
- g_flashOn 是一个全局布尔变量（1 字节）
- 当 g_flashOn == 1 时，ApplyLights 同时点亮 P1.1(NS_YELLOW) 和 P1.4(EW_YELLOW)
- 当 g_flashOn == 0 时，两个黄灯同时熄灭
- g_flashOn 翻转发生在 SystemTick2ms 中（主循环上下文），后续 ApplyLights 在同一调用中读取并输出

**是否存在一个方向黄灯亮而另一个不亮的情况？**

- g_flashOn 的翻转是原子操作（1 字节赋值）
- ApplyLights 在 g_flashOn 翻转后**同一调用中**立即读取并使用
- P1.1 和 P1.4 的赋值是两条连续指令，但都在同一函数调用中
- 两条指令之间不会发生上下文切换（SystemTick2ms 运行在主循环中，Timer0ISR 不操作 P1 口）
- 因此两个黄灯的赋值实际上在微秒级时间内完成，视觉上完全同步

**退出夜间模式的状态清理（Bug #4）：**
- Bug #4 确认退出夜间时将 g_flashOn 置 1，g_flashTicks 清零
- 退出后调用 StartPhase(PHASE_NS_GREEN)，恢复正常交通灯
- 退出瞬间不会出现黄灯残留

| 判定 | 详细说明 |
|:--:|------|
| ✅ | g_flashOn 是单一变量，两个黄灯由同一标志控制，在同一函数调用中同时赋值。不存在一个方向亮而另一个不亮的情况。Bug #4 已修复退出残留。 |

## 验收标准2: 两组数码管倒计时稳定显示，无明显闪烁、重影或乱码

### 2.1 闪烁阈值 — 125Hz 扫描频率分析

**需求：** 扫描频率 125Hz（需求规格说明书 三、性能指标）

**计算验证：**
- Timer0 中断周期：2ms（TH0=0xF8, TL0=0xCD，11.0592MHz 晶振，精确验证见 gap_analysis.md 3.1 节）
- 4 位循环扫描：每 2ms 扫描 1 位
- 完整刷新周期：4 × 2ms = 8ms
- 扫描频率：1000/8 = **125Hz**

**人眼闪烁融合阈值：** 约 50-60Hz。125Hz 远超阈值，理论上不可见闪烁。

**阻塞风险分析：**

Timer0ISR 的中断优先级高于主循环。可能阻塞 Timer0ISR 的因素：
1. **其他中断：** 本项目仅使用 Timer0 中断，无其他中断源，不存在中断嵌套阻塞
2. **主循环中关中断的代码段：** 代码审查报告未提及任何关中断（EA=0）的代码段。I2C 通信（OLED）使用软件延时（NOP），不关中断。如果有长延时函数（如 DelayMs），Timer0ISR 仍可正常触发

**潜在问题：** 如果 I2C 通信期间（OLED 刷新）Timer0ISR 被触发，Timer0ISR 会打断 I2C 时序。但 Bug #10 已解决 I2C 时序问题（使用 4 个 NOP），且 I2C 对时序要求有一定容差，短暂打断通常不会导致通信失败。

| 判定 | 详细说明 |
|:--:|------|
| ✅ | 125Hz 扫描频率远超人眼闪烁融合阈值（50-60Hz）。系统仅使用 Timer0 一个中断，无中断嵌套阻塞风险。 |

### 2.5 P0 口驱动能力 — 亮度分析

**硬件设计（02_硬件设计说明文档.md）：**
- P0.0-P0.7：段码输出，串联 220Ω 限流电阻
- 8×10kΩ 上拉电阻到 VCC
- P2.0-P2.3：位选，低电平有效，直接连接共阴数码管的公共阴极

**驱动电路分析：**

数码管段点亮时的电流路径（以 P0.0 输出 1 驱动段 A 为例）：
```
VCC(+5V) → 10kΩ(上拉) → P0.0 → 220Ω(限流) → LED段A(阳极) → LED段A(阴极) → 共阴极 → P2.x(位选,低电平) → GND
```

**电流计算：**
- LED 正向压降 Vf ≈ 2.0V（红色）
- 位选引脚 P2.x 输出低电平饱和压降 ≈ 0.3V
- 总回路电阻：R_total = 10kΩ + 220Ω = 10.22kΩ
- 单段电流：I_seg = (5V - 2.0V - 0.3V) / 10.22kΩ ≈ **0.26mA**

**8 段全亮时：**
- 总电流：8 × 0.26mA ≈ 2.1mA（通过位选引脚 P2.x）

**P0 口驱动能力评估：**
- P0 口每个引脚最大灌电流：约 10-15mA（标准 8051）
- 本项目 P0 口输出高电平（通过外部上拉提供电流），不灌电流
- 每段 0.26mA 远低于 P0 口能力上限
- P2 口位选引脚灌电流：2.1mA，远低于 P2 口能力上限（约 10-15mA）

**亮度评估：**
- 0.26mA 每段的电流对标准 LED 数码管来说**偏暗**
- 典型数码管段需要 5-10mA 才能达到正常亮度
- 动态扫描占空比 1/4（每 8ms 点亮 2ms），进一步降低平均亮度
- 平均电流：0.26mA × 1/4 = 0.065mA 每段，非常暗

**但需注意：**
- Bug #2 已标记为"已解决"，说明在 Proteus 仿真中亮度可以接受
- Proteus 仿真的 LED 亮度模型可能与实际硬件不同
- 如果需要在实物上达到正常亮度，建议将 10kΩ 上拉电阻改为更小的值（如 1kΩ 或 470Ω），或增加段驱动电路（如 74HC245 或 PNP 三极管）

**硬件设计文档中是否有段驱动电路：**
- 元器件清单中未列出 74HC245、PNP 三极管（除蜂鸣器的 NPN 外）等段驱动元件
- 仅依赖 8×10kΩ 上拉电阻 + 8×220Ω 限流电阻

| 判定 | 详细说明 |
|:--:|------|
| ⚠️ | 0.26mA/段的驱动电流偏暗，动态扫描 1/4 占空比进一步降低平均亮度。Bug #2 在 Proteus 仿真中已解决，但实物可能存在亮度不足问题。 |
| | **建议：** 若用于实物，将 P0 上拉电阻从 10kΩ 改为 1kΩ 或 470Ω（每段电流约 2.5-5mA），或增加 74HC245 作为段驱动。若仅用于 Proteus 仿真，当前设计可接受。 |

### 3.5 OLED 显示更新 — 500ms 延迟

### 4.4 行人请求的持久性 — g_pedNsReq 清零问题

**核心问题：** g_pedNsReq 在 CalcGreenTime 使用后是否被清零？如果不清零，下一轮 NS_GREEN 又会 +3 秒，导致行人请求永久生效。

**文档中的证据搜索：**
- 代码审查报告：未提及 g_pedNsReq 的清零
- Bug 日志：未提及任何与请求标志清零相关的 Bug
- 流程图：未展示清零步骤
- gap_analysis.md 4.2 节将此列为未确认问题

**风险分析：**

假设 g_pedNsReq 永不自动清零：
- 第 1 轮 NS_GREEN：g_pedNsReq=1 → CalcGreenTime +3 秒 ✅
- 第 2 轮 NS_GREEN：g_pedNsReq 仍为 1 → CalcGreenTime 又 +3 秒 ❌
- 第 3 轮 NS_GREEN：同上 ❌
- ...永久持续

**合理的清零时机：**
- 在 CalcGreenTime 读取 pedReq 后立即清零：`g_pedNsReq = 0;`
- 或在 StartPhase 中，进入 NS_GREEN 后清零

**⚠️ 如果实际代码中未清零，这是一个严重缺陷。**

| 判定 | 详细说明 |
|:--:|------|
| ❌ | **所有文档均未提及 g_pedNsReq/g_pedEwReq 的清零逻辑。** 如果确实未清零，行人请求将永久生效，导致每轮 CalcGreenTime 都 +3 秒。这是验收标准 4 的严重缺陷。 |
| | **修正建议：** 在 CalcGreenTime 函数末尾（或 StartPhase 中进入绿灯后），添加对应的请求标志清零操作：<br>`if (pedReq) { g_pedNsReq = 0; /* 或 g_pedEwReq = 0; */ }`<br>或者在 CalcGreenTime 的调用方（StartPhase/AdvancePhase）中，调用后立即清零对应方向的请求标志。 |
