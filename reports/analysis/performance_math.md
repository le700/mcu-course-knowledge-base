## 1. Timer0 定时精度

### 已知参数

- 晶振频率：\( f_{osc} = 11.0592 \text{ MHz} = 11,059,200 \text{ Hz} \)
- 机器周期：\( T_{machine} = \dfrac{12}{f_{osc}} = \dfrac{12}{11,059,200} \approx 1.0850694 \text{ } \mu s \)
- 定时器计数频率：\( f_{timer} = \dfrac{f_{osc}}{12} = 921.6 \text{ kHz} \)
- 每个计数周期：\( T_{count} = \dfrac{1}{921,600} \approx 1.08507 \text{ } \mu s \)
- 重载值：TH0 = 0xF8, TL0 = 0xCD，即 \( 0xF8CD = 63,693 \)
- 计数值：\( N = 65,536 - 63,693 = 1,843 \)

### 精确计算

定时时间：

\[
T = N \times T_{machine} = 1,843 \times \frac{12}{11,059,200}
\]

\[
T = \frac{1,843 \times 12}{11,059,200} = \frac{22,116}{11,059,200}
\]

\[
T = \frac{22,116}{11,059,200} = 0.00199975... \text{ s} = 1.99975... \text{ ms}
\]

换一种方式验证：

\[
\frac{22,116}{11,059,200} = \frac{22,116 \div 12}{11,059,200 \div 12} = \frac{1,843}{921,600} = 0.00199978... \text{ s}
\]

更精确地：

\[
1,843 \times \frac{12}{11,059,200} = \frac{22,116}{11,059,200}
\]

\[
11,059,200 \times 2 = 22,118,400
\]

\[
22,116 \div 11,059,200 = 0.001999783... \text{ s}
\]

即 \( 1,999.783 \text{ } \mu s \approx 2,000 \text{ } \mu s \)。

### 误差分析

目标值：\( 2.000 \text{ ms} = 2,000.000 \text{ } \mu s \)  
实际值：\( 1,999.783 \text{ } \mu s \)  
绝对误差：\( |2,000.000 - 1,999.783| = 0.217 \text{ } \mu s \)  
相对误差：\( \dfrac{0.217}{2,000} \times 100\% = 0.0109\% \)

**结论：** ✅ 定时误差仅约 0.217 us（0.011%），对于 2ms 定时精度完全可以忽略。该晶振频率（11.0592MHz）之所以被选为经典 51 单片机时钟频率，正是因为其分频后能产生整数波特率和精确的定时值。1,843 个计数周期恰好给出约 2ms 的定时，误差来源于 1,843 并非精确的 2,000 所需的计数数（理论值 = 2000 / 1.0850694 = 1,843.2），取整后引入 0.2 个计数周期的误差，且该误差在每个中断周期内不会累积（自动重载模式）。

## 2. 数码管扫描频率

### 计算

- Timer0 中断周期：\( 2 \text{ ms} \)
- 每次中断扫描 1 位（4 位轮转，NS十位 → NS个位 → EW十位 → EW个位）
- 每位刷新频率：\( \dfrac{1,000}{2} = 500 \text{ Hz} \)（每位每秒被点亮 500 次）
- 完整扫描周期（4 位各点亮一次）：\( 4 \times 2 \text{ ms} = 8 \text{ ms} \)
- 帧率（完整画面刷新频率）：\( \dfrac{1,000}{8} = 125 \text{ Hz} \)

### 闪烁分析

| 参数 | 值 | 说明 |
|------|-----|------|
| 帧率 | 125 Hz | 每秒完整刷新 125 次 |
| 人眼闪烁融合阈值 | 50-60 Hz | 临界闪烁频率 (CFF) |
| 安全余量 | 125/60 ≈ 2.08 倍 | 远高于阈值 |

**结论：** ✅ 125 Hz > 60 Hz，超过人眼闪烁融合阈值 2 倍以上，没有任何可见闪烁。

### 占空比与亮度分析

| 参数 | 值 |
|------|-----|
| 每位点亮时间 | ≈ 2 ms（实际因中断处理时间略短，约 1.98ms） |
| 占空比 | 1/4 = 25% |
| 每段电流（5V 供电，220Ω限流） | \( I_{seg} \approx \dfrac{5V - V_{LED}}{220\Omega} \approx \dfrac{5 - 1.8}{220} \approx 14.5 \text{ mA} \) |
| 平均每段电流 | \( 14.5 \times 0.25 \approx 3.6 \text{ mA} \) |

**P0 口驱动能力评估：**

- P0 口为开漏输出，外接 8 × 10kΩ 上拉电阻（Bug #2 已解决）
- 上拉电阻提供的高电平驱动电流：\( I_{pullup} \approx \dfrac{5V}{10k\Omega} = 0.5 \text{ mA} \)（单路）
- 当段码输出高电平（LED 熄灭）时，由 10kΩ 上拉提供高电平，仅需维持 TTL 高电平（>2.4V），0.5mA 足够
- 当段码输出低电平（LED 点亮）时，P0 口吸收电流能力约 3.2mA（标准 TTL），段码 LED 通过 220Ω 限流电阻从 VCC 经 LED 流入 P0.x 到地
- 实际吸收电流：\( I_{sink} \approx \dfrac{5V - 1.8V}{220\Omega} \approx 14.5 \text{ mA} \)
- **风险点：** 14.5mA 超过 P0 口标准 TTL 的 3.2mA 吸收能力。但在 25% 占空比下，只要峰值电流在 P0 口规格范围内（AT89C52 单引脚最大灌电流 10-15mA），即可正常工作。实际仿真中数码管亮度正常。

**结论：** ✅ 25% 占空比下，如果 P0 口灌电流能力足够（实际 Proteus 仿真验证通过），数码管亮度可接受。但在实际硬件中建议使用 74HC245 或 ULN2003 作为段码驱动缓冲。

## 3. 按键消抖时间

### KeyScan 逻辑分析

```
g_keyTicks 每 2ms 递增（SystemTick2ms 中）
g_keyTicks >= 5 时采样一次按键（即 5 × 2ms = 10ms）
DEBOUNCE_CNT = 3，需要连续 3 次采样一致才确认
总消抖时间 = 3 × 10ms = 30ms
```

### 数学推导

| 阶段 | 时间点 | 操作 |
|------|--------|------|
| 初始 | t = 0 | 按键按下，g_keyTicks 从 0 开始递增 |
| 第1次采样 | t = 10 ms | g_keyTicks = 5，采样，debounceCnt = 1 |
| 第2次采样 | t = 20 ms | g_keyTicks = 10，采样，如一致则 debounceCnt = 2 |
| 第3次采样 | t = 30 ms | g_keyTicks = 15，采样，如一致则 debounceCnt = 3，确认有效 |

**总消抖时间：30 ms**

### 与机械按键特性对比

| 参数 | 值 |
|------|-----|
| 标准机械按键抖动时间 | 5-20 ms |
| 系统消抖时间 | 30 ms |
| 安全余量 | 30 - 20 = 10 ms |

**结论：** ✅ 30ms > 20ms，消抖充分，能可靠滤除机械按键抖动。

### 潜在问题分析：按键响应延迟与快速连按

| 场景 | 分析 |
|------|------|
| 按键响应延迟 | 30ms 消抖 + 约 0.3ms 主循环扫描 = 约 30.3ms，对于人工按键（通常 > 100ms）完全可接受 |
| 快速连按识别 | 两次按键间隔需 > 30ms（消抖回零 + 重新采样3次），否则第二次被过滤 |
| 车流检测（模拟红外传感器） | 如果连续两辆车快速通过（间隔 < 30ms），只检测到一辆 |

**对于车流检测按键（模拟红外传感器）：**

需求文档（2.4 节）明确说明"P3.4/P3.5 车流检测按键按下一次，表示对应方向红外检测到一辆车"。这说明车流检测在 Proteus 仿真中是通过手动按键模拟的，人工按键间隔远大于 30ms，因此 30ms 消抖完全足够。但在实际红外传感器应用中，如果车辆以 120km/h 速度通过（约 33m/s），两车最小安全间距约 20m，时间间隔约 600ms，远大于 30ms，因此实际应用中也不会丢失检测。

**结论：** ✅ 30ms 消抖对人工按键和红外传感器实际应用均足够。

## 4. 夜间闪烁周期

### 计算

```
SystemTick2ms 中：
g_flashTicks 每 2ms 递增
g_flashTicks >= 250 时翻转 g_flashOn 并清零 g_flashTicks
```

- 闪烁半周期：\( 250 \times 2 \text{ ms} = 500 \text{ ms} \)
- 闪烁全周期：\( 2 \times 500 \text{ ms} = 1,000 \text{ ms} = 1 \text{ s} \)
- 闪烁频率：\( \dfrac{1}{1 \text{ s}} = 1 \text{ Hz} \)

**等等，重新审视：** 需求规格说明书（2.9 节）和性能指标表均写的是"夜间闪烁频率 2Hz（500ms 翻转）"。

这里存在一个定义问题：

- 如果"500ms 翻转"指的是每次翻转间隔 500ms，则亮 500ms + 灭 500ms = 1000ms 周期，频率 = 1 Hz
- 如果"2Hz"指的是亮灭各一次（即 1 秒内亮灭各 2 次），则周期 = 500ms，翻转间隔 = 250ms

根据代码逻辑 `g_flashTicks >= 250` 触发翻转：
- 在半周期翻转一次意味着：亮 250ms → 灭 250ms → 亮 250ms → ...
- 半周期 = 250ms，全周期 = 500ms，频率 = 2 Hz

**重新计算：**

| 参数 | 值 |
|------|-----|
| 翻转阈值 | g_flashTicks >= 250 |
| 翻转间隔 | 250 × 2ms = 500ms |
| 亮状态持续时间 | 500ms |
| 灭状态持续时间 | 500ms |
| 完整周期 | 500ms + 500ms = 1000ms = 1s |
| 频率 | 1 Hz |

**不一致！** 需求文档标注的是"2Hz"，但代码逻辑产生的是 1Hz。

### 深入分析

需要确认代码中 `g_flashTicks >= 250` 的阈值含义。如果 `g_flashOn` 在每次中断（2ms）中检查 `g_flashTicks`，那么：

- g_flashTicks 从 0 到 249 共 250 个 tick，250 × 2ms = 500ms
- 在 g_flashTicks = 250 时翻转，所以状态保持 500ms 后翻转

**占空比：** 50%（亮 500ms，灭 500ms）

**结论：** ⚠️ 代码实际频率为 **1 Hz**（周期 1000ms），而非需求文档声称的 2 Hz。需求文档（2.9 节）写的"夜间闪烁频率 2Hz"与代码实现存在偏差。如果需求明确要求 2Hz（即周期 500ms，亮 250ms 灭 250ms），则阈值应改为 `g_flashTicks >= 125`。但考虑到夜间黄灯闪烁通常使用 1Hz（每秒闪一次）是常见做法，且验收标准（6 条）只要求"夜间键按下后两方向黄灯闪烁"，未明确要求具体频率，因此 1Hz 可接受。

## 7. 智能配时计算

### CalcGreenTime 算法

```
sec = 12 + ownCnt * 2
if ownCnt > 6:  ownCnt = 6
if ownCnt > otherCnt + 1:  sec += 3
if pedReq:  sec += 3
if sec > 25:  sec = 25
if sec < 8:   sec = 8
```

### ownCnt 截断上限分析

需要确认代码中 ownCnt 截断的时机。根据需求文档（2.6 节）和代码审查报告（3.1 节），ownCnt 在 CalcGreenTime 内部被截断为 6，但 g_nsVehicleCnt 的实际上限为 9（代码中 `g_nsVehicleCnt < 9` 限制）。

**分析：** 这意味着即使等待队列中实际有 7-9 辆车，CalcGreenTime 也只按 6 辆计算。这是合理的工程设计——避免绿灯时间过长导致另一方向等待过久。

### 下限 8 秒触发条件分析

| 条件 | 分析 |
|------|------|
| ownCnt = 0, pedReq = 0 | sec = 12, 不小于 8 |
| ownCnt = 0, pedReq = 1 | sec = 15, 不小于 8 |
| 任何组合 | 基础值 12 已经大于 8 |

**结论：** ⚠️ 在当前算法参数下，`sec < 8` 的下限条件为**死代码**——在任何合法的 ownCnt（u8 无符号，0-6 截断范围内）和 pedReq 组合下，sec 计算结果始终 >= 12 > 8，下限永远不会被触发。

**但这不是 bug：** 下限 8 秒是防御性编程，为未来可能修改基础绿灯时间（例如将 12 改为 6）或调整算法参数预留的安全边界。保留该代码是良好的工程实践。

### 算术溢出检查

- ownCnt 截断后最大为 6，sec 最大计算路径：12 + 6×2 + 3 + 3 = 30，截断后为 25
- 所有中间值均为 u8 类型（0-255），不会溢出
- 最终结果限制在 8-25，符合需求

**结论：** ✅ 智能配时算法逻辑正确，边界行为符合预期，无算术溢出风险。

## 9. 中断响应时间

### Timer0 中断延迟分析

**硬件响应延迟：**

| 阶段 | 时间 | 说明 |
|------|------|------|
| 中断请求采样 | 0 | 每个机器周期的 S5P2 采样中断标志 |
| 中断确认 | 1-2 机器周期 | 查询中断优先级和使能位 |
| LCALL 到 ISR | 2 机器周期 | 硬件自动压栈 PC 并跳转 |
| 硬件响应总计 | 3-8 机器周期 | 取决于当前指令执行状态 |

**最坏情况：** 当前正在执行 RETI 或访问 IE/IP 寄存器的指令时，需要额外等待一个指令周期。

| 乐观情况 | 3 × 1.085us = 3.26 us |
|----------|------------------------|
| 典型情况 | 5 × 1.085us = 5.43 us |
| 最坏情况 | 8 × 1.085us = 8.68 us |

### Timer0ISR 执行时间估算

```
Timer0ISR 主要操作：
1. 重载 TH0/TL0                  ~2 机器周期
2. 消隐（关位选）                 ~3 机器周期
3. 段码写入 P0                   ~2 机器周期
4. 位选切换 (switch-case)        ~4 机器周期
5. 中断返回 RETI                 ~2 机器周期
─────────────────────────────────────────
总计                            ~13-15 机器周期
```

实际估算（含编译器优化）：
- 约 15-20 机器周期 ≈ 16.3-21.7 us

### 总中断响应时间

| 指标 | 值 |
|------|-----|
| 硬件响应延迟 | 3.3 - 8.7 us |
| ISR 执行时间 | 16.3 - 21.7 us |
| **总中断处理时间** | **19.6 - 30.4 us** |
| 中断间隔 | 2,000 us |
| 中断占用率 | 30.4/2000 = 1.52% |

**结论：** ✅ 中断处理时间（< 31 us）远小于中断间隔（2,000 us），CPU 有 98.5% 的时间用于主循环。不会出现中断重入或丢失问题。

## 10. 主循环执行时间

### 主循环结构分析

```c
while (1) {
    KeyScan();           // 6 个按键轮询
    HandleKeys();        // 按键状态处理
    OLED_UpdateScreen(); // 条件刷新（500ms 一次）
    SystemTick2ms();     // 实际上此函数在别处调用
}
```

### 各函数执行时间估算

**KeyScan()：**

| 操作 | 估算周期 |
|------|----------|
| 6 个按键电平读取（P3 口） | 6 × 2 = 12 |
| 去抖计数器和状态机 | ~15 |
| 总计 | ~27 机器周期 ≈ 29.3 us |

**HandleKeys()：**

| 操作 | 估算周期 |
|------|----------|
| 紧急模式判断 | ~5 |
| 夜间模式判断 | ~5 |
| 行人请求处理 | ~10 |
| 总计 | ~20 机器周期 ≈ 21.7 us |

**OLED_UpdateScreen()（500ms 刷新一次时）：**

| 操作 | 估算周期 |
|------|----------|
| I2C 通信（SSD1306 全屏刷新） | ~50,000 机器周期 ≈ 54.3 ms |

**注意：** OLED_UpdateScreen 在 500ms 刷新一次时，I2C 写入是阻塞式的。54.3ms 的理论估算与 Bug #5 中提到的"全屏写入耗时约 40ms"基本吻合。但该函数仅在 g_oledDue 为真时执行，且执行后即清除标志，因此每 500ms 只阻塞一次主循环，其余 499 次主循环均为快速通过。

**OLED_UpdateScreen()（非刷新时，即大部分主循环周期）：**

| 操作 | 估算周期 |
|------|----------|
| 检查 g_oledDue 标志 | ~2 |
| 直接返回 | ~2 |
| 总计 | ~4 机器周期 ≈ 4.3 us |

### 主循环总执行时间

| 场景 | 估算周期 | 时间 |
|------|----------|------|
| 普通循环（无 OLED 刷新） | ~51 机器周期 | ~55.4 us |
| OLED 刷新循环（每 500ms 一次） | ~50,051 机器周期 | ~54.3 ms |

### 主循环频率

| 场景 | 频率 |
|------|------|
| 普通循环 | 1/55.4us ≈ 18,000 Hz |
| 受 OLED 刷新影响时 | 每 500ms 中有 1 次 ~54ms 的阻塞 + 约 499 次 ~55us 的循环 |

**结论：** ✅ 主循环在绝大部分时间以约 18 kHz 的频率运行，响应速度远超按键输入（人工 > 100ms）和系统事件（2ms 中断）的需求。OLED 刷新每 500ms 阻塞约 54ms（占 10.8%），不会影响系统整体响应。

### 发现的问题

1. **夜间闪烁频率不一致（⚠️）：** 需求文档标注"2Hz"，代码实际产生 1Hz（周期 1000ms，亮 500ms 灭 500ms）。需确认需求意图。若要求 2Hz，应将 `g_flashTicks` 阈值从 250 改为 125。

2. **智能配时下限 8 秒为死代码（⚠️）：** 在当前算法参数下，`sec < 8` 永远不会被触发。这是防御性编程，保留可接受，但建议在代码注释中说明。

3. **数码管 P0 口驱动能力（潜在风险）：** 25% 占空比下，P0 口灌电流约 14.5mA，接近 AT89C52 引脚极限。实际硬件中建议使用驱动缓冲器。
