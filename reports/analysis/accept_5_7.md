### 5.1 优先方向选择算法：SelectEmergencyDirection

**文档来源：** 需求规格说明书 2.8 节，Bug#7，代码审查报告 2.1 节

**算法要素（从文档还原）：**

| 因素 | 数据来源 | 含义 |
|------|---------|------|
| 因素 A：红外检测方向 | 最近一次 P3.4/P3.5 按键触发方向 | 最近检测到车辆的方向 |
| 因素 B：等待车辆数 | g_nsVehicleCnt vs g_ewVehicleCnt | 红灯期间累计的等待车辆数 |
| 因素 C：当前通行方向 | g_currentPhase (NS_GREEN/NS_YELLOW/EW_GREEN/EW_YELLOW) | 当前正在通行的方向 |

**权重与冲突解决：**

- **判定：❌ 无法完整验证（算法细节缺失）**

代码审查报告仅确认 `SelectEmergencyDirection` 函数存在（main.c 行 911-961），并声明"综合考虑红外检测方向、等待车辆数和当前通行方向三个因素"。Bug#7 确认该函数是为解决"方向选择不明确"而新增的。然而，**任何一份文档均未描述以下关键细节**：

1. **三个因素各自的权重**：未说明哪个因素优先。例如，红外检测 = NS、等待较多 = EW、当前 = NS 时，最终选择 NS 还是 EW？
2. **冲突解决规则**：因素指向不同方向时，是多数投票（2:1 胜出）、加权打分、还是硬性优先级链？
3. **红外优先**：需求文档说"最近一次车辆检测方向"，但未声明红外检测是否具有否决权或最高优先级。
4. **等待车辆数相同**：g_nsVehicleCnt == g_ewVehicleCnt 时如何处理？是 fallback 到当前方向，还是 fallback 到红外方向？
5. **函数返回值**：未说明返回值是一个枚举值（DIR_NS / DIR_EW）还是直接修改 g_emergencyDir 全局变量。

**验证结论：**

| 测试场景 | 预期行为 | 可验证性 |
|---------|---------|---------|
| 三个因素都指向 NS | 选择 NS | ✅ 高置信度（无冲突） |
| 三个因素都指向 EW | 选择 EW | ✅ 高置信度（无冲突） |
| 红外=NS, 等待=EW, 当前=NS | 未知 | ❌ 算法未文档化 |
| 红外=EW, 等待=NS, 当前=EW | 未知 | ❌ 算法未文档化 |
| 红外=NS, 等待=EW, 当前=EW | 未知 | ❌ 算法未文档化 |
| 两个方向等待车辆数相同 | 未知 | ❌ 算法未文档化 |

**建议修正：**
- 在代码审查报告或需求文档中补充 `SelectEmergencyDirection` 的伪代码或决策表。
- 推荐优先级：红外检测方向 > 等待车辆数 > 当前方向，或采用计分制（红外 3 分、等待差 2 分、当前 1 分，总分高者胜出）。

### 5.3 数码管 "EE" 显示

**文档来源：** 需求规格说明书 2.8 节，代码审查报告 2.1 节，硬件设计说明文档第三节

**验证：**

硬件设计使用 4 位数码管（P2.0-P2.3 位选），其中 NS 方向占 2 位（十位 + 个位），EW 方向占 2 位（十位 + 个位）。

**判定：⚠️ 通过但存在歧义**

| 问题 | 分析 |
|------|------|
| 4 位都显示 "E" 还是仅 2 位？ | 需求文档只写"数码管显示 EE"，未明确是 4 位还是 2 位。从代码审查报告 `UpdateDisplay` 函数（行 686-740）的描述推断，紧急模式下 `g_dispBuf` 被赋值为 4 个 "E" 字符的段码（即 NS 两位 + EW 两位都显示 "E"）。这是最合理的实现，因为 4 位扫描是统一进行的。 |
| "E" 的段码值 | 对于共阴数码管，段码 "E" = 0x79（A=1, D=1, E=1, F=1, G=1）。但代码审查报告未给出具体段码值。 |

**确认：** 4 位数码管均显示 "E"（NS 十位 = E, NS 个位 = E, EW 十位 = E, EW 个位 = E），视觉上呈现为 "EE EE" 或 "EE"（取决于观察者视角）。需求文档期望的 "EE" 应理解为两个方向各显示 "EE"，即 4 位全部为 "E"。

**建议修正：** 在需求文档中明确"两组数码管（NS 和 EW）均显示 EE，共 4 位 E 字符"。

### 5.4 蜂鸣器输出

**文档来源：** 需求规格说明书 2.8 节，硬件设计说明文档第五节（蜂鸣器驱动电路），软件流程图（SystemTick2ms）

**验证：**

硬件设计文档确认：
- P2.7 = 1 → NPN 导通 → 蜂鸣器鸣响
- P2.7 = 0 → NPN 截止 → 蜂鸣器静音

**判定：⚠️ 通过但存在歧义**

| 问题 | 分析 |
|------|------|
| 持续响还是间歇响？ | 需求文档只写"蜂鸣器端口 P2.7 输出有效"，未明确持续/间歇。从软件流程图看，`ApplyLights` 在每次 SystemTick2ms 中都被调用，如果紧急模式下在 `ApplyLights` 中设置 `P2.7 = 1`，则蜂鸣器为持续鸣响。如果通过 `SystemTick2ms` 中的闪烁逻辑控制（类似夜间黄灯闪烁），则为间歇鸣响。代码审查报告未提及蜂鸣器的闪烁控制，因此推断为 **持续鸣响**（g_emergency=1 时 P2.7 恒为 1）。 |
| 是否会产生听觉疲劳？ | 持续鸣响在仿真中可接受，但实际应用中通常使用间歇鸣响（如 500ms 周期）以增强警示效果。 |

**建议修正：** 在需求文档中明确"蜂鸣器以 500ms 周期间歇鸣响"或"蜂鸣器持续鸣响"，消除歧义。如果需要间歇鸣响，应在 `SystemTick2ms` 中增加蜂鸣器闪烁逻辑（类似 `g_flashOn` 机制）。

### 5.6 紧急模式退出后状态恢复

**文档来源：** 需求规格说明书 2.8 节，软件流程图（HandleKeys），代码审查报告 2.1 节

**验证：**

从 HandleKeys 流程图：

```
F{g_emergency == 1?}
F -- 否 --> H[StartPhase NS_GREEN]
```

**判定：⚠️ 通过但存在改进空间**

| 场景 | 分析 |
|------|------|
| 进入紧急前 = EW_GREEN, 剩余 10 秒 | 退出紧急后强制进入 `PHASE_NS_GREEN`，EW 方向剩余 10 秒丢失。NS 方向获得新的绿灯时间（默认 12 秒 + 智能配时）。 |
| 是否合理？ | 设计上合理。强制回到 NS_GREEN 是一个**确定性行为**，避免了状态恢复不一致的风险。但 EW 方向的等待时间被浪费，可能造成不公平。 |
| 是否有更好的方案？ | 是。可以在进入紧急模式前保存当前相位和剩余秒数：`g_savedPhase = g_currentPhase; g_savedSec = g_phaseSec;`，退出时恢复到保存的状态。这样 EW_GREEN 剩余的 10 秒不会被浪费。 |

**建议修正：**

```c
// 建议在 HandleKeys 中增加状态保存/恢复
// 进入紧急时：
if (g_emergency == 1) {
    g_savedPhase = g_currentPhase;   // 保存当前相位
    g_savedSec   = g_phaseSec;       // 保存剩余秒数
    SelectEmergencyDirection();
}
// 退出紧急时：
else {
    // 方案A（当前实现）：强制 NS_GREEN
    StartPhase(PHASE_NS_GREEN);
    
    // 方案B（建议）：恢复到保存的状态
    // g_currentPhase = g_savedPhase;
    // g_phaseSec = g_savedSec;
    // ApplyLights();
    // 如果保存的相位是绿灯，恢复智能配时后的时间
}
```

**注意：** 需求文档未明确要求状态恢复，当前实现已满足验收标准"再次按下紧急键退出，恢复正常循环"。⚠️ 标记仅表示存在改进空间。

### 7.5 人机协作反思中的仿真问题

**文档来源：** 人机协作反思文档，Agent MCP Skills 文档
