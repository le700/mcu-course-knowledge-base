### 验证点 2.3.1：中文字模覆盖范围

**问题：** `oled_zh_font.h`（126行）是否包含所有必要中文字符的字模？每个16x16中文字模占32字节，126行代码能容纳多少字？

| 来源 | 字符 | 数量 |
|------|------|:--:|
| 四行标签 | 模、式、等、待、车、数、红、外 | 8 |
| 模式值（正常） | 正、常 | 2 |
| 模式值（紧急） | 紧、急 | 2 |
| 模式值（夜间） | 夜、间 | 2 |
| **合计** | | **14** |

代码审查报告（`03_代码审查报告.md`）第91行指出：
> "OLED 全部中文字模占约 544 bytes code 空间"

按每个16x16中文字模占用32字节计算：
- 544 bytes / 32 bytes/字 = **17 个中文字符**

这意味着字模文件中实际包含了 **17 个中文字符**，比所需的14个多出 **3 个**。这3个额外字符可能用于扩展显示（如"行人"、"通行"等），或包含占位符/备用字符。多出的3个字符不影响功能正确性。

**126行文件长度分析：**

典型的 `oled_zh_font.h` 结构：
```
行 1-4:   文件头注释 + #ifndef 宏保护
行 5-6:   #include 或空行
行 7-8:   数组声明（如 code unsigned char font_xxx[] = {）
行 9-24:  字模数据（每字32字节，若每行16字节则2行/字，若每行8字节则4行/字）
...
行 125:   结束 };
行 126:   #endif
```

若每行16个十六进制字节（常见格式），每字2行数据 + 1行注释 = 3行/字。17字 × 3 = 51行数据 + 约15行头尾 = 约66行。126行显著多于66行，说明可能采用每行8字节的稀疏格式（每字4行数据），或包含较多空行/注释分隔。17字 × 4行数据 = 68行 + 15行头尾 = 83行，仍不足126行。

**结论：** 126行容纳17个中文字符绰绰有余。14个必要字符全部被覆盖的概率极高（544字节 = 17字 > 14字）。但**无法从现有文档中逐字验证字模表中是否确实包含全部14个目标字符**，因为 `oled_zh_font.h` 源文件不在工作目录中。

| 判定 | 说明 |
|:--:|------|
| **⚠️** | 从代码审查报告的544字节数据量推算，17字字模库足以覆盖14字需求。但无法在缺乏源文件的情况下逐一核对每个字符的字模数据是否确实存在且正确。**建议补充确认 `oled_zh_font.h` 中确实包含模、式、等、待、车、数、红、外、正、常、紧、急、夜、间这14个字。** |

### 验证点 2.3.4：模式显示

**问题：** 需求说"模式只显示'正常、紧急、夜间'"。代码中 `g_oledModeCache` 是否只在这三种模式之间切换？

**分析：**

代码审查报告（`03_代码审查报告.md`）第77行：
> "使用缓存变量（g_oledModeCache 等）避免重复刷新"

```
紧急按键按下 → g_emergency 翻转
  ├─ g_emergency == 1 → SelectEmergencyDirection → 显示"紧急"
  └─ g_emergency == 0 → StartPhase(NS_GREEN)
  
夜间按键按下 → g_night 翻转
  ├─ g_night == 1 → 显示"夜间"
  └─ g_night == 0 → StartPhase(NS_GREEN)
```

流程图显示紧急模式处理后有 `return`（第75-76行），夜间模式处理后有 `return`（第79-80行），这是互斥的。因此系统只能处于三种状态之一：

`g_oledModeCache` 用于缓存当前显示的模式字符串，与此三种状态一一对应。

**结论：** 逻辑上，`g_oledModeCache` 确实只在"正常、紧急、夜间"三种模式之间切换，不存在第四种模式。状态机保证了互斥性。

### 验证点 2.3.5：OLED 刷新机制

**问题：** `OLED_UpdateScreen` 是否只在 `g_oledDue` 标志置位时才刷新？刷新间隔是否满足500ms？

**分析：**

`07_软件流程图_mermaid.md` 中的主程序流程图（图一）显示：

```
g_oledDue != 0? 
  ├─ 是 → g_oledDue = 0 → OLED_UpdateScreen → 回到主循环
  └─ 否 → 回到主循环
```

**确认：** `OLED_UpdateScreen` 确实只在 `g_oledDue` 非零时被调用，且调用后立即清零 `g_oledDue`。

`07_软件流程图_mermaid.md` 中的 SystemTick2ms 流程图（图二）显示 `g_oledDue` 被置位的两个条件：

1. **定时刷新（第50-51行）：**
   ```
   g_oledTicks++ → oledTicks >= 250? → g_oledDue = 1
   ```
   每个 SystemTick2ms 调用增加一次 `g_oledTicks`，250 次 × 2ms = **500ms**。与需求规格说明书中"OLED 刷新间隔 500ms"一致。

2. **相位切换时刷新（第43-46行）：**
   ```
   phaseSec == 0? → AdvancePhase → g_oledDue = 1
   ```
   当倒计时归零、相位切换时立即触发一次刷新，不等500ms定时。这保证了模式切换时 OLED 能即时更新。

Bug #5（`04_bug解决日志.md` 第57-63行）确认了：
> "改用局部刷新 + 缓存比较，只更新变化的区域，避免全屏重绘"

**结论：** 刷新机制符合需求：500ms 定时刷新 + 相位切换时即时刷新，且使用局部刷新 + 缓存比较避免闪屏。

| 判定 | 说明 |
|:--:|------|
| **✅** | 流程图确认 `g_oledDue` 作为刷新门控，500ms 定时刷新（250 × 2ms = 500ms）与需求一致。Bug #5 确认了局部刷新 + 缓存比较机制。 |

### 验证点 2.3.6：I2C 通信时序

**问题：** OLED使用P1.6(SCL)和P1.7(SDA)，I2C地址0x78。Bug #10修复了I2C时序不稳。4个NOP延时是否足够？

**分析：**

硬件设计文档（`02_硬件设计说明文档.md` 第83-84行）：
> "P1.6 | OLED_SCL | I2C 时钟线"  
> "P1.7 | OLED_SDA | I2C 数据线"

第161行：
> "OLED 选 SSD1306，I2C 地址 0x78"

Bug #10（`04_bug解决日志.md` 第102-107行）：
> "使用 4 个 NOP 作为 I2cDelay，保证最小延时足够"

**时序计算（11.0592MHz 晶振）：**

| 参数 | 值 |
|------|-----|
| 晶振频率 | 11.0592 MHz |
| 机器周期 | 12 / 11.0592 MHz ≈ 1.085 μs |
| 1个 NOP | 1 机器周期 ≈ 1.085 μs |
| 4个 NOP | 4 × 1.085 μs ≈ **4.34 μs** |

**I2C 标准模式（100kHz）时序要求：**

| 参数 | 最小值 | 4 NOP 实际值 | 满足？ |
|------|:--:|:--:|:--:|
| SCL 低电平保持时间 | 4.7 μs | 4.34 μs | **否（差 0.36 μs）** |
| SCL 高电平保持时间 | 4.0 μs | 4.34 μs | 是 |
| SDA 建立时间 | 250 ns | 4.34 μs | 是 |
| SDA 保持时间 | 0 ns | 4.34 μs | 是 |

**分析：** 4个NOP产生的4.34μs延时在SCL低电平保持时间上**略低于I2C标准模式的最低要求**（4.7μs）。但需要注意：

1. NOP 只是延时的一部分，I2C 函数中还有端口赋值、函数调用开销等，实际延时略大于4.34μs。
2. SSD1306 的实际时序容差通常比标准更宽松。
3. Bug #10 描述为"已解决"，说明在实际 Proteus 仿真中 4 个 NOP 已足够稳定。

**结论：** 4个NOP的延时在严格意义上略低于I2C标准，但考虑实际指令开销和SSD1306的容差，在Proteus仿真中已足够。如果移植到实际硬件且遇到通信异常，建议增加至5-6个NOP。

| 判定 | 说明 |
|:--:|------|
| **⚠️** | 4个NOP（约4.34μs）略低于I2C标准模式SCL低电平最低要求（4.7μs），但考虑到函数调用开销和SSD1306的容差，在Proteus仿真中已确认稳定（Bug #10已解决）。实际硬件上若出现通信异常，建议增加至5-6个NOP。 |

### 验证点 2.4.3：按键消抖对车流检测的影响

**问题：** 消抖时间30ms。如果用户在30ms内连按两次（模拟两辆车快速通过），第二次会被消抖过滤掉吗？

**分析：**

`07_软件流程图_mermaid.md` 图七 和 `03_代码审查报告.md` 第71-73行：

按键扫描周期：每 5 × 2ms = **10ms** 扫描一次（`keyTicks >= 5` 触发）。
消抖确认：连续 **3次** 采样一致 → 3 × 10ms = **30ms** 消抖窗口。
边沿检测：**上升沿**（`prev==0 && stable!=0`）触发，即按键释放瞬间。

**快速连按场景分析：**

```
时间轴：  0ms    10ms   20ms   30ms   40ms   50ms   60ms
按键1：  按下────────────释放──────────────────────────
按键2：                              按下────────────释放
```

如果两次按键间隔 > 30ms（消抖窗口），两个按键事件都会被正常检测到。

如果两次按键间隔 < 30ms：

```
时间轴：  0ms    10ms   20ms   30ms   40ms   50ms
按键1：  按下────释放
按键2：           按下────────────释放
```

第二次按键发生在第一次按键的消抖窗口内，消抖计数器会被重置，导致两次按键被合并或第二次被丢失。

**对需求语义的影响：** 需求原文"按键按下一次，表示对应方向红外检测到一辆车"——这个语义本身就是"一次按下一辆车"，而非"快速连按模拟多辆车"。在Proteus仿真中，按键模拟的是红外传感器信号，实际红外传感器不会有"机械抖动"问题，但消抖是必要的，因为按键本身有机械抖动。

**结论：** 30ms消抖窗口的设计对模拟"一辆车"的语义是合理的。如果两次按键间隔 < 30ms，第二次可能被过滤，但这对于"按一次表示一辆车"的语义而言是合理的，因为用户需要有意地、明确地按每次键。

| 判定 | 说明 |
|:--:|------|
| **✅** | 30ms消抖窗口对"按下一次表示检测到一辆车"的语义是合理的。快速连按（<30ms间隔）可能被过滤，但这不是需求的本意——需求意图是每次按键代表一辆车，而非支持极高速连续输入。 |

### 验证点 2.5.1：红灯判断逻辑

**问题：** `RegisterVehicle` 中如何判断当前方向是否为红灯？逻辑是否正确？

**分析：**

`07_软件流程图_mermaid.md` 图五定义了四个相位状态：

```
NS_GREEN → NS_YELLOW → EW_GREEN → EW_YELLOW → (循环)
```

交通灯互锁规则（需求2.1）：某方向绿灯或黄灯时，另一方向保持红灯。因此：

| 当前相位 | NS方向状态 | EW方向状态 |
|------|:--:|:--:|
| NS_GREEN | 绿灯（通行） | 红灯 |
| NS_YELLOW | 黄灯（通行） | 红灯 |
| EW_GREEN | 红灯 | 绿灯（通行） |
| EW_YELLOW | 红灯 | 黄灯（通行） |

`RegisterVehicle` 中判断"该方向是否为红灯"的逻辑应为：

- **NS方向车辆检测：** `g_phase` 不等于 `NS_GREEN` 且不等于 `NS_YELLOW` → 即 `g_phase == EW_GREEN || g_phase == EW_YELLOW` 时 NS 为红灯 → 计入等待队列
- **EW方向车辆检测：** `g_phase` 不等于 `EW_GREEN` 且不等于 `EW_YELLOW` → 即 `g_phase == NS_GREEN || g_phase == NS_YELLOW` 时 EW 为红灯 → 计入等待队列

代码审查报告（`03_代码审查报告.md` 第58行）确认了：
> "仅当该方向为红灯时才计入等待队列"

**结论：** 红灯判断逻辑正确——只有在该方向相位为对向通行（绿/黄）时，本方向才为红灯，车辆才计入等待队列。

| 判定 | 说明 |
|:--:|------|
| **✅** | 红灯判断逻辑正确：NS方向在 EW_GREEN/EW_YELLOW 时为红灯，EW方向在 NS_GREEN/NS_YELLOW 时为红灯。代码审查报告确认了这一逻辑。 |

### 验证点 2.5.2：等待队列上限不一致

**问题：** 代码审查报告说 `g_nsVehicleCnt < 9` 限制，但需求2.6说"上限6辆"。9 vs 6 不一致！

**分析：**

代码审查报告（`03_代码审查报告.md` 第59行）：
> "等待队列上限为 9（代码中 g_nsVehicleCnt < 9 限制）"

需求2.6（`01_项目需求规格说明书.md` 第37-40行）：
> "每辆等候车辆 +2 秒（上限 6 辆，最多 +12 秒）"

代码审查报告（`03_代码审查报告.md` 第63行）：
> "CalcGreenTime 中 ownCnt 上限 6（防止溢出）"

以及 `03_代码审查报告.md` 第90行：
> "CalcGreenTime 中 ownCnt 上限 6，但实际等待可能更多"

**分析这两层上限：**

| 上限位置 | 变量 | 上限值 | 作用 |
|------|------|:--:|------|
| `RegisterVehicle` | `g_nsVehicleCnt` / `g_ewVehicleCnt` | **9** | 等待队列最大容量 |
| `CalcGreenTime` | `ownCnt` 参数 | **6** | 参与配时计算的最大车辆数 |

**这意味着：**
- 等待队列可以累计到 **9 辆车**（第7-9辆车仍被记录在队列中）
- 但在计算绿灯时间时，`CalcGreenTime` 只取前 **6 辆**（`ownCnt = min(actual, 6)`）
- 第7-9辆车虽然"在等待"，但不会增加绿灯时间

**这是否合理？** 需求2.6明确说"上限6辆"，指的是配时计算的上限。需求2.5只说"计入等待车数"，没有明确等待队列的上限。9是等待队列的实现上限，6是配时计算的上限。两个上限不一致是**设计上的有意区分**，但需求文档中仅明确了配时上限（6），未明确等待队列上限（9）。

代码审查报告（`03_代码审查报告.md` 第89行）也注意到了这个问题：
> "g_nsVehicleCnt 上限为 9，而 OLED 显示为 2 位数 | 可改为 99 或保持不变（9 已够用）"

**结论：** 等待队列上限（9）与配时计算上限（6）不一致。这并非bug，而是有意设计：等待队列可容纳9辆，但配时只取前6辆。需求文档仅明确了配时上限6，未对等待队列上限作出规定。但从一致性角度看，建议统一为6或明确文档中的差异。

| 判定 | 说明 |
|:--:|------|
| **⚠️** | 等待队列上限（9）与需求2.6的配时上限（6）不一致。这不是bug，而是有意设计——等待队列可容纳9辆，但配时仅使用前6辆（`ownCnt` capped at 6）。需求文档未明确等待队列的上限，但从规范一致性角度，建议统一或明确标注差异。 |

### 验证点 2.5.3：等待队列清零

**问题：** 进入该方向绿灯后，等待队列是否被清零？

**分析：**

`07_软件流程图_mermaid.md` 图二（SystemTick2ms）：
```
phaseSec-- → phaseSec == 0? → AdvancePhase → g_oledDue = 1
```

`07_软件流程图_mermaid.md` 图五（状态机）：
```
NS_GREEN → NS_YELLOW → EW_GREEN → EW_YELLOW → NS_GREEN
```

代码审查报告（`03_代码审查报告.md` 第22行）：
> "相位控制 | StartPhase / AdvancePhase | 584-631 | 正常"

`StartPhase` 在进入新相位时被调用，`AdvancePhase` 在相位切换时被调用。

**清零逻辑的推断：** 当进入 NS_GREEN 时，NS方向的等待车辆（`g_nsVehicleCnt`）应当被清零，因为这些车辆正在被服务。清零应该在 `StartPhase(PHASE_NS_GREEN)` 或 `CalcGreenTime` 之后执行。

代码审查报告（`03_代码审查报告.md` 第39-45行）显示：

- `CalcGreenTime` 的参数 `ownCnt` 来源正是 `g_nsVehicleCnt` / `g_ewVehicleCnt`

**但现有文档中未明确提及等待队列清零逻辑。** `StartPhase` 和 `AdvancePhase` 的职责描述为"正常"（代码审查报告第22行），但未说明具体包含哪些操作。

代码审查报告（`03_代码审查报告.md` 第89行）的评论：
> "g_nsVehicleCnt 上限为 9"

这个上限注解暗示了 `g_nsVehicleCnt` 在 `RegisterVehicle` 中递增，但未提及清零位置。

**结论：** 从逻辑必要性推断，等待队列应在进入该方向绿灯时清零。但现有文档中**未明确记录清零逻辑的位置**（是在 `StartPhase`、`AdvancePhase` 还是 `CalcGreenTime` 中？）。需要查看源码确认。

| 判定 | 说明 |
|:--:|------|
| **⚠️** | 等待队列必须在进入绿灯时清零，否则会导致配时异常。现有文档未明确记录清零逻辑的位置。**建议在源码中确认 `StartPhase` 或 `AdvancePhase`（main.c 第584-631行）是否包含 `g_nsVehicleCnt = 0` / `g_ewVehicleCnt = 0` 的清零操作。** |

### 验证点 2.5.4：黄灯期间的边界情况

**问题：** 如果车辆在黄灯期间被检测到，应该计入等待吗？

**分析：**

需求原文（`01_项目需求规格说明书.md` 第33-34行）：
> "如果该方向当前为绿灯或黄灯，则只计入累计车流量，不计入等待队列。"

**明确结论：黄灯期间不计入等待队列。**

以 NS 方向为例的相位时间线：

```
NS_GREEN(12-25s) → NS_YELLOW(3s) → EW_GREEN(12-25s) → EW_YELLOW(3s) → ...
                   ↑
                   NS方向黄灯 = 即将变红灯
                   此时NS车辆检测到 → 不计入等待队列
```

**逻辑检验：** 黄灯期间车辆仍可通行（需求2.1），所以不计入等待队列是合理的。但黄灯只有3秒，很快将变为红灯，此时检测到的车辆确实"即将等待"。需求明确选择了"不计入"，这可能是为了简化逻辑——避免在黄灯3秒的短暂窗口内做出特殊处理。

`RegisterVehicle` 中判断红灯的逻辑应为：
- NS方向：`g_phase != NS_GREEN && g_phase != NS_YELLOW` → 即仅 EW_GREEN 和 EW_YELLOW 时计入等待
- NS_YELLOW 时检测到 NS 车辆 → 不计入等待（符合需求）

代码审查报告（`03_代码审查报告.md` 第58行）：
> "仅当该方向为红灯时才计入等待队列"

**结论：** 需求明确规定黄灯期间不计入等待队列，这与代码审查报告的描述一致。黄灯的设计意图是"过渡期"，车辆仍可通行。

| 判定 | 说明 |
|:--:|------|
| **✅** | 需求明确"绿灯或黄灯时只计累计流量"，黄灯期间不计入等待队列。代码审查报告确认了"仅当红灯时才计入"的逻辑。这一设计合理——黄灯期间车辆仍可通行。 |
