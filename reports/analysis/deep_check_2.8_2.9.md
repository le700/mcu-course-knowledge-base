### 1.3 SelectEmergencyDirection 三因素算法分析

**Bug #7 原文：** "增加 SelectEmergencyDirection 函数，综合考虑红外检测方向、等待车辆数和当前通行方向三个因素。"

**三个因素定义：**

| 因素 | 数据来源 | 含义 |
|------|----------|------|
| 因素1：红外检测方向 | 最近一次 P3.4/P3.5 按键触发方向 | 紧急车辆从哪个方向来 |
| 因素2：等待车辆较多方向 | `g_nsVehicleCnt` vs `g_ewVehicleCnt` | 哪个方向积压更多车 |
| 因素3：当前通行方向 | `g_phase` 当前值 | 哪个方向正在放行 |

**算法推断（基于代码审查报告的模块结构描述）：**

`SelectEmergencyDirection` 位于 main.c 第 911-961 行（与 `RegisterVehicle` 同属"车辆检测"模块），约 50 行代码。合理的逻辑推断如下：

1. **优先级规则（推测）：** 红外检测方向 > 等待车辆较多方向 > 当前通行方向
   - 因素1（红外）是最直接指示紧急车辆所在方向的信号，优先级最高
   - 因素2（等待车数）反映道路拥堵程度，优先级次之
   - 因素3（当前通行方向）作为兜底判断

2. **多数投票规则（推测）：** 三个因素中取多数指向的方向
   - 若 2:1 或 3:0 → 选择多数方向
   - 若 1:2 同理

**举例推演（用户给出的场景）：**

> 红外检测到 NS 方向有车、EW 方向等待车辆更多（5 vs 2）、当前是 EW_GREEN

| 因素 | 指向方向 | 
|------|----------|
| 因素1（红外） | NS |
| 因素2（等待车数） | EW（5 > 2） |
| 因素3（当前相位） | EW（EW_GREEN） |

- 多数投票：EW 2 票，NS 1 票 → 选择 **EW** 方向
- 优先级规则：红外最高 → 选择 **NS** 方向

**判定：** ⚠️ **无法精确验证 — 源代码不可用**

代码审查报告仅描述了函数的存在和位置，未给出具体算法逻辑。Bug 解决日志只说"综合考虑三个因素"，未说明权重或优先级。这是文档与代码之间的关键信息缺口。

**强烈建议：** 
1. 补充 `SelectEmergencyDirection` 的算法伪代码至代码审查报告
2. 明确三个因素的优先级关系和冲突解决策略
3. 增加针对性的测试用例（见表末附录）

### 1.5 蜂鸣器输出 — P2.7 驱动验证

**硬件电路（硬件设计文档第五节）：**

```
P2.7 ───[500Ω]───┐
                  │
               ┌──┴──┐
               │ NPN  │  (8050 或通用 NPN)
               └──┬──┘
                  │
               ┌──┴──┐
               │BUZZER│  (有源蜂鸣器)
               └──┬──┘
                  │
                  VCC (+5V)
```

**电路分析：**
- P2.7 = 1（高电平）→ 经过 500Ω 限流电阻 → NPN 基极 → 基极电流 `I_B ≈ (5V - 0.7V) / 500Ω ≈ 8.6mA`
- NPN 导通 → 集电极-发射极形成低阻抗通路 → 蜂鸣器两端电压 ≈ VCC - V_CE(sat) ≈ 5V - 0.3V = 4.7V
- 蜂鸣器得电 → 鸣响
- P2.7 = 0 → NPN 截止 → 蜂鸣器无电流 → 静音

**电路正确性：** ✅ **电路设计正确，P2.7=1 蜂鸣器响**

**蜂鸣器行为：**

需求文档："蜂鸣器端口 P2.7 输出有效" — 未明确持续鸣响还是有间歇。

代码审查报告和流程图均未描述蜂鸣器在紧急模式下的具体控制逻辑（是否通过 `g_flashOn` 产生间歇效果，还是持续导通）。

**推断：** 由于 `ApplyLights` 在 SystemTick2ms 中每次都被调用，蜂鸣器控制可能也在其中。如果在紧急模式下 `P2.7 = 1` 持续输出，则蜂鸣器持续鸣响。

**判定：** ⚠️ **无法确认间歇/持续行为** — 需求文档未明确，代码审查报告未描述

**建议：** 需求应明确蜂鸣器是持续鸣响还是间歇鸣响（如 500ms 周期），并在代码中实现。

| 场景 | 进入紧急前相位 | 退出紧急后相位 | 是否合理？ |
|------|----------------|----------------|------------|
| 场景 A | NS_GREEN | NS_GREEN | ✅ 巧合一致 |

退出紧急模式后**总是**进入 NS_GREEN，忽略了进入紧急模式前的系统状态。如果紧急前 EW 方向正在放行，退出后 EW 方向的车辆被"插队"。

**与 Bug #4 的类比：**

- `g_emergencyDir`：退出后应清零
- `g_phaseSec`：是否需要重新计算？
- 蜂鸣器 P2.7：退出后应确保置 0（静音）

**修正建议：**

1. **保存进入紧急前的相位**：`g_prevPhase = g_phase`（在 SelectEmergencyDirection 之前）
2. **退出时恢复**：`StartPhase(g_prevPhase)` 而非硬编码 `StartPhase(PHASE_NS_GREEN)`
3. **清理紧急状态**：`g_emergencyDir = 0`、`P2.7 = 0`（静音）

### 1.7 紧急模式与智能配时的关系

**流程图证据（图二 SystemTick2ms）：**

```

K -- 否 --> Q[g_oledDue = 1]

| 变量 | 紧急模式下的行为 | 判定 |
|------|------------------|------|
| `g_secondTicks` | 不递增 | ✅ |

**判定：** ✅ **紧急模式下倒计时正确暂停，退出后重新计算配时**

这说明行人按键处理确实在 HandleKeys 中，位于夜间键检查之后。当 `g_emergency == 1` 时，`J --是--> K([返回])` 直接返回，跳过所有后续处理（包括行人键）。

**但存在一个关键问题：** 流程图显示车辆检测键（NS_CAR/EW_CAR）在紧急键检查**之前**：

```
B{NS_CAR / EW_CAR?}
B -- 是 --> C[RegisterVehicle 方向]
B -- 否 --> D{紧急按键按下?}
C --> D
```

这意味着：**在紧急模式下，按下车辆检测键仍会触发 RegisterVehicle**。这可能导致：
- 紧急模式下 `g_nsVehicleCnt` / `g_ewVehicleCnt` 被意外修改
- 退出紧急后 CalcGreenTime 使用错误的等待车辆数

**判定：** ⚠️ **车辆检测在紧急模式下未被屏蔽**

**修正建议：** 在 `RegisterVehicle` 调用前增加 `!g_emergency` 条件判断，或将其移至紧急键检查之后。

### 2.2 进入夜间模式 — 黄灯同步闪烁机制

**流程图证据（图三 HandleKeys）：**

```

M --> N{g_night == 1?}

**黄灯闪烁机制（图二 SystemTick2ms）：**

```
G{g_night?}
G -- 是 --> H[g_flashTicks++]
H --> I{flashTicks >= 250?}
I -- 是 --> J[g_flashOn 翻转]
I -- 否 --> K{!emergency && !night?}
J --> K
```

**时间计算：**
- 每次 SystemTick2ms 调用时 `g_flashTicks++`
- 到达 250 时翻转 `g_flashOn`，即 250 x 2ms = **500ms 周期**
- 2Hz 闪烁频率，与需求一致

**ApplyLights 中的夜间模式输出（推测）：**

`ApplyLights` 位于 main.c 第 633-684 行。夜间模式下应对两方向输出黄灯。关键在于 `g_flashOn` 信号：

- `g_flashOn = 1` → 两方向黄灯同时亮（NS_YELLOW=1, EW_YELLOW=1），其他灯灭
- `g_flashOn = 0` → 所有灯灭

**同步性验证：** `g_flashOn` 是单一全局变量，两方向黄灯共用同一信号，因此**必然同步**（同时亮、同时灭）。

**判定：** ✅ **两方向黄灯同步闪烁，500ms 周期，逻辑正确**

**Bug #4 的退出修复：** 退出夜间模式时 `g_flashOn = 1` 重置，确保黄灯不处于熄灭状态。流程图 `N --否--> O[StartPhase NS_GREEN]` 中应包含此逻辑。

**与退出紧急模式相同的硬编码问题：** 退出夜间后总是进入 NS_GREEN，如果夜间前是 EW_GREEN，方向被反转。

**修正建议：** 保存进入夜间前的相位，退出时恢复。

### 2.6 夜间模式中按其他键

**HandleKeys 流程分析：**

```
B{NS_CAR / EW_CAR?}  ← 第一步：车辆检测
C --> D{紧急按键按下?}  ← 第二步：紧急键
D -- 否 --> ...        ← 紧急键未按下
J -- 否 --> L{夜间按键按下?}  ← 第三步：夜间键
```

**场景推演：**

| 在夜间模式下按... | 流程行为 | 判定 |
|-------------------|----------|------|
| 车辆检测键 (NS_CAR/EW_CAR) | 第一步匹配 → RegisterVehicle 执行 → 车辆计数增加 | ❌ 不合理 |
| 紧急键 | 第二步匹配 → g_emergency 翻转 → 进入/退出紧急模式 | ✅ 合理（见 2.7 节） |
| 行人键 (NS_PEDES/EW_PEDES) | 不在流程图中，位于夜间键之后 → 被跳过 | ✅ 合理 |

**核心问题：** 夜间模式下车辆检测键仍被处理。夜间模式下黄灯闪烁，理论上不应累积车辆计数。

**修正建议：** 在 RegisterVehicle 调用前增加 `!g_night && !g_emergency` 条件。

### 2.7 夜间模式与紧急模式的交叉（关键缺陷）

**场景 A：夜间模式中按紧急键**

```
初始状态：g_night=1, g_emergency=0

SystemTick2ms → HandleKeys:
  步骤1: NS_CAR/EW_CAR? → 否
  步骤2: 紧急按键按下? → 是
    → g_emergency 翻转 = 1
    → g_emergency==1 → SelectEmergencyDirection
    → g_emergency 为 1 → 返回  ← g_night 仍为 1！
```

**此时状态：** `g_night=1` 且 `g_emergency=1`

**SystemTick2ms 行为：**
```
G{g_night?} → 是 → g_flashTicks++  ← 夜间闪烁仍在运行！
K{!emergency && !night?} → !emergency 为假 → 跳过计时
ApplyLights → 紧急模式灯光输出（优先方向绿灯）
```

**退出紧急模式（再按紧急键）：**

```
HandleKeys:
  步骤2: 紧急按键按下? → 是
    → g_emergency 翻转 = 0
    → g_emergency==0 → StartPhase(NS_GREEN)
    → g_emergency 为 0 → 检查夜间键
    → 夜间键未按下 → 返回  ← g_night 仍为 1！
```

**此时状态：** `g_night=1`，`g_emergency=0`

**SystemTick2ms 行为：**
```
G{g_night?} → 是 → 夜间闪烁继续
K{!emergency && !night?} → !night 为假 → 跳过计时
ApplyLights → 夜间模式 → 黄灯闪烁
```

**结论：** 退出紧急模式后，系统自动回到夜间模式。**这是正确的行为还是 Bug？** 从用户视角看，如果用户本意是"先退出紧急模式，再手动退出夜间模式"，这是合理的。但如果用户期望"退出紧急模式即恢复正常交通灯"，则这是一个 Bug。

**需求文档未对此场景做出规定。**

---

**场景 B：紧急模式中按夜间键**

```
初始状态：g_emergency=1, g_night=0

HandleKeys:
  步骤1: NS_CAR/EW_CAR? → 否
  步骤2: 紧急按键按下? → 否（没有按紧急键）
  → J{g_emergency?} → 是 → 返回  ← 夜间键被完全跳过！
```

**结论：** 紧急模式下夜间键被完全忽略。`g_night` 保持为 0。

**判定：** ✅ **紧急模式下夜间键被正确屏蔽，不存在交叉污染**

---

**场景 C：完整交叉测试矩阵**

| 初始状态 | 按下键 | 最终 g_night | 最终 g_emergency | 系统行为 | 判定 |
|----------|--------|-------------|------------------|----------|------|
| 正常 | 紧急 | 0 | 1 | 紧急模式 | ✅ |
| 正常 | 夜间 | 1 | 0 | 夜间模式 | ✅ |
| 夜间 | 紧急 | **1（未清除）** | 1 | 紧急+夜间闪烁 | ⚠️ |
| 夜间 | 夜间 | 0 | 0 | 正常（NS_GREEN） | ✅ |
| 紧急 | 夜间 | **0（被屏蔽）** | 1 | 紧急模式 | ✅ |
| 紧急 | 紧急 | 0 | 0 | 正常（NS_GREEN） | ✅ |
| 夜间+紧急 | 紧急 | **1（未清除）** | 0 | 夜间模式 | ⚠️ |
| 夜间+紧急 | 夜间 | **0** | **1（未清除）** | 紧急模式 | ⚠️ |

**场景 C 第 3 行（夜间 → 紧急）的详细分析：**

1. 初始：夜间模式，黄灯闪烁
2. 按紧急键 → `g_emergency=1`, `g_night` 仍为 1
3. 系统同时处于两种模式：`ApplyLights` 输出紧急灯光，但 `g_flashTicks` 仍在递增
4. 再按紧急键 → `g_emergency=0`, `g_night` 仍为 1 → 系统回到夜间模式
5. 用户需要再按一次夜间键才能完全退出 → 需要 3 次按键操作

**场景 C 第 8 行（夜间+紧急 → 按夜间键）的详细分析：**

1. 初始：夜间模式中按了紧急键，现在 `g_night=1, g_emergency=1`
2. 按夜间键 → HandleKeys 中紧急键未按下 → `g_emergency=1` → 直接返回
3. 夜间键被完全忽略！`g_night` 仍为 1
4. 用户无法退出夜间模式，除非先退出紧急模式

**判定：** ❌ **存在严重的交叉模式状态污染 Bug**

**修正建议：**

```c
// 在 HandleKeys 中进入紧急模式时，如果当前处于夜间模式，先退出夜间
if (g_emergency == 1) {
    if (g_night) {
        g_night = 0;          // 清除夜间标志
        g_flashOn = 1;        // 重置闪烁
        g_flashTicks = 0;
    }
    SelectEmergencyDirection();
}
// 进入夜间模式时，如果当前处于紧急模式，先退出紧急
if (g_night == 1) {
    if (g_emergency) {
        g_emergency = 0;      // 清除紧急标志
        P2_7 = 0;             // 关闭蜂鸣器
        StartPhase(PHASE_NS_GREEN);
    }
}
```

或者更简洁的方案：**紧急模式与夜间模式互斥**，进入一个时自动退出另一个。
