## 二、硬件设计 vs 需求的一致性

### 2.1 MCU 平台兼容性 🟢轻微

**需求：** STC89C52RC / AT89C52  
**硬件设计：** AT89C52（Proteus 标准模型）

**分析：**  
- Proteus 仿真中 AT89C52 与 STC89C52RC 在指令集层面完全兼容（均为 8051 内核）
- 本项目未使用 STC 特有功能（如片上 EEPROM、看门狗、PCA 等），因此用 AT89C52 模型仿真 STC89C52RC 程序没有问题
- HEX 文件可正常加载，代码量 3618 bytes 远小于 8KB Flash 上限
- **风险：** 如果实际打板使用 STC89C52RC，需注意 STC 的复位电平要求（高电平复位）与 AT89C52 一致，但 STC 对复位电路阻容值有更严格要求

---

### 2.2 数码管驱动能力 🟢轻微

**需求：** 两组二位共阴数码管（2×7SEG-MPX2-CC）  
**硬件设计：** P2.0-P2.3 四位选，P0 口输出段码

**分析：**  
- 7SEG-MPX2-CC 是 2 位共阴数码管，内部 2 个数字的段码引脚并联，各有独立的共阴极
- 2 个 × 2 位 = 4 位，P2.0-P2.3 正好 4 根位选线，一一对应
- 两个数码管单元的段码引脚需要在 Proteus 中并联后再接到 P0 口，文档未显式说明此连线，但属于常规接法
- P0 口已确认有 8×10kΩ 上拉电阻（Bug #2 已解决），驱动能力满足要求
- 位选为低电平有效，与 7SEG-MPX2-CC 的共阴极特性一致

## 三、性能指标 vs 实现

### 3.1 Timer0 定时值计算 ✅ 正确

**需求：** 中断周期 2ms，扫描频率 125Hz  
**代码：** TH0=0xF8, TL0=0xCD

**验证计算：**  
```
晶振频率：11.0592 MHz
机器周期：11.0592 / 12 = 0.9216 MHz
周期时间：1 / 0.9216 MHz ≈ 1.085 μs
2ms 所需计数：2000 / 1.085 ≈ 1843
重载值：65536 - 1843 = 63693 = 0xF8CD
```

**结论：** TH0=0xF8, TL0=0xCD 精确对应 2ms 定时。扫描频率 = 1/(2ms×4) = 125Hz。完全正确。

---

### 3.2 按键消抖 30ms ✅ 正确

**分析：**  
```
g_keyTicks 每 2ms 递增一次
KeyScan 触发条件：g_keyTicks >= 5 → 5×2ms = 10ms 间隔
DEBOUNCE_CNT = 3 → 3 次采样确认
总消抖时间：3 × 10ms = 30ms
```

**结论：** 与需求中"30ms（3次采样确认）"完全一致。

---

### 3.3 夜间闪烁 500ms ✅ 正确

**分析：**  
```
g_flashTicks 每 2ms 递增一次
翻转条件：g_flashTicks >= 250 → 250×2ms = 500ms
翻转周期：500ms × 2 = 1s（闪烁频率 1Hz）
```

**结论：** 500ms 翻转周期正确，但需求说"500ms 周期"而实际闪烁频率是 1Hz（500ms 亮 + 500ms 灭 = 1000ms 周期）。需求中的"500ms 周期"应理解为"500ms 翻转周期"（即 2Hz 闪烁频率）。代码实现与需求性能指标表中"2 Hz（500ms 翻转）"一致。

---

### 3.4 OLED 刷新间隔 500ms ✅ 正确

**分析：**  
```
g_oledTicks 每 2ms 递增一次
刷新条件：g_oledTicks >= 250 → 250×2ms = 500ms
```

**结论：** 与需求中"OLED 刷新间隔 500ms"一致。

---

### 3.5 绿灯时长范围 8~25 秒 ✅ 正确

**分析：** CalcGreenTime 中基础 12 秒 + 每车 2 秒 (上限 6 辆 → +12 秒) + 车多 +3 秒 + 行人 +3 秒，最终 clamp 到 8~25 秒。与需求一致。

## 四、潜在逻辑漏洞

### 4.1 智能配时"多 2 辆以上"的语义歧义 🟡中等

**需求：** "本方向车比对方多 2 辆以上 → 额外 +3 秒"  
**代码：** `ownCnt > otherCnt + 1`

**分析：**  

| 条件 | ownCnt=3, otherCnt=1 | ownCnt=2, otherCnt=0 | ownCnt=3, otherCnt=0 |
|------|----------------------|----------------------|----------------------|
| ownCnt > otherCnt+1 | 3 > 2 → True | 2 > 1 → True | 3 > 1 → True |
| ownCnt > otherCnt+2 | 3 > 3 → False | 2 > 2 → False | 3 > 2 → True |

- 代码条件 `ownCnt > otherCnt + 1` 等价于 `ownCnt >= otherCnt + 2`，即"本方向比对方多至少 2 辆"
- 中文"多 2 辆以上"存在歧义：可理解为"至少多 2 辆"（与代码一致），也可理解为"超过 2 辆"即"至少多 3 辆"
- 在当前语境下，代码的实现（至少多 2 辆）更符合中文日常表达习惯，且与需求文档撰写的通常意图一致

**结论：** 实现基本合理，但建议在需求文档中明确为"多 2 辆及以上"或"多至少 2 辆"以消除歧义。

---

### 4.2 行人请求同时触发冲突 🔴严重

**场景：** 两个方向的行人键在同一扫描周期内被按下

**分析：**  
- `HandleKeys` 函数会依次处理 NS 行人键和 EW 行人键（Bug #8 确认行人处理在 HandleKeys 中）
- 假设当前 NS 绿灯，两方向行人键同时按下：
  - NS 行人键：本向绿灯 → 保证剩余时间 >= 6 秒（延长）
  - EW 行人键：对向绿灯 → 缩短 NS 剩余时间至 5 秒（缩短）
  - 哪个生效取决于代码中的处理顺序，后处理的会覆盖前者
- 流程图中 HandleKeys 部分（图三）甚至没有展示行人按键的处理逻辑，存在文档性缺失

**风险：**  
- 如果 NS 行人键后处理，NS 绿灯至少 6 秒，但 EW 行人期望缩短被忽略
- 如果 EW 行人键后处理，NS 绿灯被缩短至 5 秒，但 NS 行人期望延长被忽略
- 实际上某一方的行人请求被静默丢弃

**建议：** 增加冲突仲裁逻辑，例如：当两个方向同时有行人请求时，优先处理先到达的一方，或两个请求都接受（绿灯时间取 max(6, 5) = 6 秒）。

---

### 4.3 紧急模式方向选择优先级不透明 🔴严重

**需求：** "系统根据最近一次车辆检测方向、等待车辆数量和当前通行方向选择一个方向优先放行"  
**代码：** `SelectEmergencyDirection` 函数（911-961 行），Bug #7 修复后"综合考虑三个因素"

**问题：**  
- 三个因素的具体优先级权重在文档中完全没有说明
- Bug #7 的描述仅为"综合考虑红外检测方向、等待车辆数量和当前通行方向三个因素"，没有给出算法细节
- 流程图（图三）中 `SelectEmergencyDirection` 是一个黑盒节点，没有任何内部逻辑

**场景分析：**  
假设以下情况同时发生：
- 最近一次车辆检测是 NS 方向
- EW 方向等待车辆有 5 辆，NS 方向只有 1 辆
- 当前通行方向是 NS

三个因素指向不同方向，最终选择哪个？文档无法回答。

**风险：** 如果方向选择逻辑不合理，可能导致紧急车辆被引导到错误方向，完全违背紧急模式的初衷。

**建议：** 文档化 `SelectEmergencyDirection` 的完整决策逻辑，并补充到流程图中。

---

### 4.4 紧急模式退出后丢失原状态 🟡中等

**需求：** "再次按下紧急键退出，恢复正常循环"  
**代码：** 退出紧急模式后调用 `StartPhase(PHASE_NS_GREEN)`（代码审查报告 2.1 节确认）

**分析：**  
- 系统总是从 NS_GREEN 开始恢复，不保留进入紧急前的相位和倒计时状态
- 如果进入紧急前是 EW_GREEN 还剩 8 秒，退出后直接跳到 NS_GREEN 重新开始，相当于丢失了 EW 方向的通行权
- 这种"硬重置"策略虽然简化了实现，但可能造成交通流不公平

**判断：** 对于教学级项目，这种简化是可接受的。需求只要求"恢复正常循环"，未要求保留状态。但应在文档中明确说明这一行为。

---

### 4.5 紧急模式与夜间模式交叉状态污染 🔴严重

**场景：** 在夜间模式下按下紧急键，再退出紧急模式

**分析（基于流程图图三）：**  
1. 初始状态：g_night = 1, g_emergency = 0
2. 按下紧急键 → g_emergency = 1 → SelectEmergencyDirection → 检查 g_emergency → 为真 → 直接返回（跳过夜间键检查）
3. **关键：g_night 仍然是 1，未被清除！**
4. 再次按下紧急键 → g_emergency = 0 → StartPhase(PHASE_NS_GREEN) → g_oledDue = 1 → 检查 g_emergency → 为假 → 检查夜间键 → 夜间键未被按下 → 返回
5. 结果：g_night = 1 但 g_emergency = 0，系统处于正常相位但夜间标志仍然置位

**后果：**  
- SystemTick2ms 中 `g_night` 为真 → g_flashTicks 持续递增、黄灯闪烁逻辑激活
- 条件 `!emergency && !night` 为假 → g_secondTicks 不递增 → 倒计时不工作
- 但 ApplyLights 可能按正常模式输出灯光（因为 g_emergency=0 且相位已重置为 NS_GREEN）
- **系统进入一个既非正常也非夜间的混乱状态**

**建议：** 在进入紧急模式时，显式清除 g_night 标志；或在紧急模式退出后，检查并恢复 g_night 相关的状态一致性。

---

### 4.6 车辆检测按键快速连按的漏检风险 🟢轻微

**场景：** 用户快速连按车辆检测键（模拟多辆车连续通过）

**分析：**  
- 消抖逻辑使用 3 次采样确认（30ms），每次上升沿只触发一次
- 如果两次按键间隔小于 30ms（消抖窗口），第二次按键的上升沿可能被消抖逻辑吞掉
- 实际上，人类的按键速度很难达到 30ms 以内（约 33 次/秒），所以正常情况下不会漏检
- 但在 Proteus 仿真中，可以通过鼠标快速点击（或脚本）触发，理论上存在漏检可能

**结论：** 对于实际使用场景，30ms 消抖是合理的，漏检风险极低。属于设计取舍而非 bug。

---

### 4.7 g_secondTicks 在模式切换后可能产生不精确的首秒 🟢轻微

**分析：**  
- g_secondTicks 在夜间/紧急模式期间不递增
- 退出这些模式时，g_secondTicks 未被重置（Bug #4 只重置了 g_flashOn 和 g_flashTicks）
- 如果 g_secondTicks 在进入模式前是 400，退出后从 400 继续计数，则首次 phaseSec 递减只需 (500-400)×2ms = 200ms 而非 1000ms
- 这会导致模式切换后的第一个倒计时秒偏短（最多偏短近 1 秒）

**建议：** 在退出夜间/紧急模式时，将 g_secondTicks 清零。

## 七、整体评价

### 7.1 系统能否完成 P-B-5# 的设计目的？

**结论：基本可以，但存在若干风险点。**

**正面因素：**  
- 代码审查报告确认全部 9 项功能需求均已实现，编译 0 Error 0 Warning
- 10 个 Bug 已全部修复，覆盖了数码管重影、按键误触发、状态残留、OLED 闪屏等关键问题
- 性能指标（2ms 定时、125Hz 扫描、30ms 消抖、500ms 闪烁）全部严格验证通过
- 代码量 3618 bytes，RAM 71.6 bytes，资源占用合理
- 状态机设计清晰，正常循环、紧急模式、夜间模式三级优先级层次分明

**负面因素：**  
- 紧急模式方向选择逻辑不透明（最大的黑盒）
- 紧急/夜间模式交叉可能导致状态污染
- 行人请求同时触发无冲突处理
- 流程图文档存在重要遗漏

### 7.2 最大的风险点

**按严重程度排序：**

| 排名 | 风险 | 严重度 | 理由 |
|------|------|--------|------|
| 1 | 紧急模式方向选择不透明 | 🔴严重 | 如果算法有误，紧急车辆可能被引导到错误方向，核心功能失效 |
| 2 | 紧急/夜间模式交叉状态污染 | 🔴严重 | 可能导致系统进入不可预期的混乱状态，需通过状态机严格验证 |
| 3 | 行人请求同时触发冲突 | 🔴严重 | 一方请求被静默丢弃，验收标准 4 可能在特定条件下不满足 |
| 4 | HandleKeys 流程图缺失行人处理 | 🔴严重 | 文档与代码不一致，误导维护者 |
| 5 | 紧急模式退出后状态丢失 | 🟡中等 | 设计取舍，但需在文档中明确说明 |
| 6 | OLED 显示内容未确认 | 🟡中等 | 功能存在但细节不可验证 |
| 7 | "--" 和 "EE" 显示实现未确认 | 🟡中等 | 同上 |

### 7.3 总结

该系统作为一个教学级项目（51 单片机课程设计），整体完成度较高。代码审查评分 9/10 是合理的。但从"需求 vs 实现"的全局视角看，存在以下根本性问题：

1. **文档与代码脱节：** 流程图无法完整反映代码逻辑（行人处理完全缺失），是最大的文档质量问题
2. **关键算法黑盒化：** 紧急方向选择和冲突处理等核心决策逻辑未文档化，导致可维护性和可验证性不足
3. **状态机边界条件：** 夜间/紧急模式之间的交叉状态转换缺乏充分测试和设计验证

**建议的优先级修复顺序：**  
1. 立即补充 HandleKeys 流程图中行人处理分支
2. 文档化 SelectEmergencyDirection 的完整决策算法
3. 添加紧急模式进入时清除夜间标志的逻辑
4. 添加行人请求冲突仲裁逻辑
5. 补充 OLED 显示内容和特殊段码定义的文档
